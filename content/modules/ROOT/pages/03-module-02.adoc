= Module 2: Convert the model-serving workloads
:source-highlighter: rouge
:toc: macro
:toclevels: 2

This is the heart of the migration. Your cluster runs four models: two on KServe Serverless mode, one on ModelMesh, and one already on KServe RawDeployment. The first three are the work -both Serverless and ModelMesh are removed in 3.x, and if you disable those components while the models still depend on them, the models stop serving. The RawDeployment model is already 3.x-compatible and you leave it alone.

So you convert *first*. In this module you'll convert the two Serverless InferenceServices to RawDeployment using the backup-and-recreate pattern the admission webhook forces, then migrate the ModelMesh model to a single-model KServe RawDeployment. Only after this is done -in Module 3 -will you disable the components.

== Learning objectives

By the end of this module, you'll be able to:

* Enumerate InferenceServices by deployment mode
* Convert a Serverless ISVC to RawDeployment with backup-and-recreate (the webhook blocks in-place edits)
* Migrate a ModelMesh model to a single-model KServe RawDeployment ISVC
* Verify every model is RawDeployment and `Ready` before touching the components

== Exercise 2.1: Enumerate what needs converting

List every InferenceService that is *not* already RawDeployment:

[.console-input]
[source,role="execute"]
----
oc get isvc -A -o json | jq -r '.items[]
  | select((.metadata.annotations."serving.kserve.io/deploymentMode" // "") != "RawDeployment")
  | "\(.metadata.namespace)\t\(.metadata.name)\t\(.metadata.annotations."serving.kserve.io/deploymentMode")"'
----

[.console-output]
----
ml-project-a   my-serverless-isvc   Serverless
ml-project-b   another-sl-model     Serverless
ml-project-c   my-modelmesh-isvc    ModelMesh
----

Two Serverless and one ModelMesh -all three must be migrated. The already-RawDeployment ISVC (`tinyllama-1b-cpu` in `ml-project-raw`) is filtered out by the query above; it needs no work.

=== Verify

[.console-input]
[source,role="execute"]
----
oc get isvc -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}={.metadata.annotations.serving\.kserve\.io/deploymentMode}{"\n"}{end}'
----

✓ You see two `Serverless` and one `ModelMesh` ISVC

== Exercise 2.2: Convert the first Serverless ISVC

[WARNING]
====
You cannot convert a Serverless ISVC with an in-place `oc patch`. The KServe admission webhook rejects it: `deploymentMode cannot be changed from 'Serverless' to 'RawDeployment'`. You must back up the spec, delete the ISVC, and recreate it with the new annotation.
====

Convert `my-serverless-isvc` in `ml-project-a`. First, back up the spec with server-managed fields stripped:

[.console-input]
[source,role="execute"]
----
NS=ml-project-a; NAME=my-serverless-isvc
oc get isvc "$NAME" -n "$NS" -o yaml \
  | yq eval 'del(.metadata.resourceVersion, .metadata.uid, .metadata.creationTimestamp, .metadata.generation, .metadata.managedFields, .status)' - \
  > "/tmp/isvc-${NS}-${NAME}.yaml"
----

Flip the deployment mode in the backup, delete the old ISVC, and recreate it:

[.console-input]
[source,role="execute"]
----
NS=ml-project-a; NAME=my-serverless-isvc
yq -i '.metadata.annotations."serving.kserve.io/deploymentMode" = "RawDeployment"' "/tmp/isvc-${NS}-${NAME}.yaml"
oc delete isvc "$NAME" -n "$NS"
oc apply -f "/tmp/isvc-${NS}-${NAME}.yaml"
----

=== Verify

Wait for the recreated ISVC to come back as RawDeployment and Ready (the model reloads, so give it a couple of minutes):

[.console-input]
[source,role="execute"]
----
oc get isvc my-serverless-isvc -n ml-project-a \
  -o custom-columns='NAME:.metadata.name,MODE:.metadata.annotations.serving\.kserve\.io/deploymentMode,STATUS:.status.deploymentMode,READY:.status.conditions[?(@.type=="Ready")].status'
----

[.console-output]
----
NAME                 MODE            STATUS          READY
my-serverless-isvc   RawDeployment   RawDeployment   True
----

✓ `MODE` and `STATUS` are both `RawDeployment` and `READY` is `True`

== Exercise 2.4: Understand the ModelMesh model

ModelMesh is *multi-model serving* -one ServingRuntime hosts many models, keyed by a storage path. 3.x has no ModelMesh, so the model must be re-deployed as a *single-model* KServe RawDeployment ISVC backed by a single-model ServingRuntime.

Inspect the ModelMesh ISVC and its multi-model runtime:

[.console-input]
[source,role="execute"]
----
oc get isvc my-modelmesh-isvc -n ml-project-c -o yaml | yq '.spec'
echo "--- multi-model ServingRuntime ---"
oc get servingruntime ovms-mm -n ml-project-c -o jsonpath='multiModel={.spec.multiModel}{"\n"}'
----

[.console-output]
----
predictor:
  model:
    modelFormat:
      name: onnx
      version: "1"
    runtime: ovms-mm
    storage:
      key: pvc-models
      path: mobilenet
--- multi-model ServingRuntime ---
multiModel=true
----

This is a MobileNetV2 image classifier in *ONNX* format, served by OpenVINO Model Server (OVMS) in ModelMesh multi-model mode. The model lives on a PVC (`model-store`) under the OVMS layout `mobilenet/1/`. To migrate it you re-deploy the *same model* on a *single-model* OVMS ServingRuntime as a KServe RawDeployment ISVC, pointing `storageUri` at the same PVC path. ONNX is a first-class format for both ModelMesh OVMS and single-model KServe OVMS, so this is a true re-host -no model conversion, no data movement.

Confirm the model file is present on the PVC:

[.console-input]
[source,role="execute"]
----
oc -n ml-project-c get pvc model-store -o jsonpath='PVC: {.status.phase}{"\n"}'
----

=== Verify

[.console-input]
[source,role="execute"]
----
oc get servingruntime ovms-mm -n ml-project-c -o jsonpath='{.spec.multiModel}{"\n"}'
----

✓ Output is `true` -confirming this is a ModelMesh multi-model runtime

== Exercise 2.5: Re-host the model on KServe RawDeployment

The model-store PVC is `ReadWriteOnce` (the default on `gp3-csi` and most cloud block-storage classes). RWO means only *one* node can attach the volume at a time. The ModelMesh runtime pod currently holds it; if you create the new KServe RawDeployment ISVC while ModelMesh is still running, the new predictor pod is very likely to land on a *different* node and hang with:

----
Multi-Attach error for volume "pvc-...": Volume is already used by pod(s) modelmesh-serving-ovms-mm-...
----

Release the PVC first by scaling the ModelMesh ServingRuntime to zero. The ISVC stays — you'll delete it cleanly in Exercise 2.6 — but its pod is gone, so the volume detaches and the new KServe pod can mount it on whatever node it's scheduled to.

[.console-input]
[source,role="execute"]
----
oc -n ml-project-c patch servingruntime ovms-mm --type=merge -p '{"spec":{"replicas":0}}'
----

Wait for the ModelMesh pod to terminate so the PVC actually detaches:

[.console-input]
[source,role="execute"]
----
oc -n ml-project-c wait --for=delete pod -l name=modelmesh-serving-ovms-mm --timeout=120s
----

[NOTE]
====
On a `ReadWriteMany` (RWX) storage class -EFS, CephFS, NFS -multiple nodes can attach the same volume at once and this scale-down is not strictly required. The lab uses RWO because it's the default on AWS gp3, so this step is what makes the migration work on the lab's storage class.
====

Now create a *single-model* OVMS ServingRuntime and a RawDeployment ISVC that reads the same ONNX model from the same PVC. Apply both:

[.console-input]
[source,role="execute"]
----
oc apply -f - <<'EOF'
apiVersion: serving.kserve.io/v1alpha1
kind: ServingRuntime
metadata:
  name: ovms-raw
  namespace: ml-project-c
  labels:
    opendatahub.io/dashboard: "true"
  annotations:
    openshift.io/display-name: "OVMS single-model (KServe)"
spec:
  multiModel: false
  supportedModelFormats:
    - name: onnx
      version: "1"
      autoSelect: true
  protocolVersions:
    - v2
    - grpc-v2
  containers:
    - name: kserve-container
      image: openvino/model_server:latest
      args:
        - --model_name={{.Name}}
        - --model_path=/mnt/models
        - --port=8001
        - --rest_port=8888
        - --file_system_poll_wait_seconds=0
        - --target_device=CPU
      ports:
        - containerPort: 8888
          protocol: TCP
      resources:
        requests:
          cpu: "500m"
          memory: 1Gi
        limits:
          cpu: "2"
          memory: 2Gi
---
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: mobilenet-onnx
  namespace: ml-project-c
  annotations:
    serving.kserve.io/deploymentMode: "RawDeployment"
    openshift.io/display-name: mobilenet-onnx
  labels:
    opendatahub.io/dashboard: "true"
spec:
  predictor:
    minReplicas: 1
    model:
      modelFormat:
        name: onnx
        version: "1"
      runtime: ovms-raw
      storageUri: pvc://model-store/mobilenet
      resources:
        requests:
          cpu: "500m"
          memory: 1Gi
        limits:
          cpu: "2"
          memory: 2Gi
EOF
----

Wait for the new RawDeployment ISVC to come up (the storage-initializer copies the model from the PVC, then OVMS loads it):

[.console-input]
[source,role="execute"]
----
oc get isvc mobilenet-onnx -n ml-project-c \
  -o custom-columns='NAME:.metadata.name,MODE:.status.deploymentMode,READY:.status.conditions[?(@.type=="Ready")].status'
----

Re-run until `READY` is `True`.

=== Verify

[.console-input]
[source,role="execute"]
----
oc get isvc mobilenet-onnx -n ml-project-c -o jsonpath='mode={.status.deploymentMode} ready={.status.conditions[?(@.type=="Ready")].status}{"\n"}'
----

[.console-output]
----
mode=RawDeployment ready=True
----

✓ The migrated `mobilenet-onnx` ISVC is serving on RawDeployment

== Troubleshooting

*Issue*: `oc apply` of the recreated ISVC fails with a `deploymentMode` webhook error.

*Solution*: Confirm you edited the *backup file* annotation to `RawDeployment` (Exercise 2.2 step 2) and that you deleted the old ISVC before re-applying. The webhook only blocks *changing* mode on an existing object; a fresh create with `RawDeployment` is accepted.

*Issue*: The recreated ISVC sits `READY=False` for several minutes.

*Solution*: RawDeployment pulls and loads the model fresh. Watch the predictor pod: `oc get pods -n <ns> -l serving.kserve.io/inferenceservice=<name>`. A CPU model can take a few minutes. Check pod logs if it never becomes ready.

*Issue*: You can't delete the multi-model ServingRuntime -"still in use".

*Solution*: Delete the ModelMesh ISVC first (Exercise 2.5). The runtime can only be removed once no ISVC references it.
