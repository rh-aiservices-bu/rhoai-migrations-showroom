= Module 2: Assess the cluster
:source-highlighter: rouge
:toc: macro
:toclevels: 2

Before you change anything, you need to know what the upgrade will break. Red Hat ships an assessment tool, `rhai-cli`, that inspects the cluster against a target version and reports every component, workload, and dependency that needs attention.

In this module you'll deploy the assessment tool, run it against your {source-version} cluster targeting {target-version}, learn to read the results, and -critically -learn why a clean assessment is *not* the same as an empty cluster.

== Learning objectives

By the end of this module, you'll be able to:

* Deploy and run the `rhai-cli` assessment tool against a live cluster
* Read a `rhai-cli lint` report and identify blocking versus advisory findings
* Inventory the model-serving and workbench workloads directly
* Explain why the assessment is not a substitute for a workload inventory

== Exercise 2.2: Deploy the assessment tool

`rhai-cli` runs from a pod inside the cluster so it can read the OLM CSVs, the DSC/DSCI, and workload custom resources cluster-wide. Create a project and deploy it.

[.console-input]
[source,role="execute"]
----
oc new-project rhai-migration
----

[.console-input]
[source,role="execute"]
----
oc apply -f - <<'EOF'
apiVersion: v1
kind: PersistentVolumeClaim
metadata: { name: rhai-cli-backup, namespace: rhai-migration }
spec:
  accessModes: ["ReadWriteOnce"]
  resources: { requests: { storage: 5Gi } }
---
apiVersion: apps/v1
kind: StatefulSet
metadata: { name: rhai-cli, namespace: rhai-migration }
spec:
  serviceName: rhai-cli
  replicas: 1
  selector: { matchLabels: { app: rhai-cli } }
  template:
    metadata: { labels: { app: rhai-cli } }
    spec:
      containers:
        - name: rhai-cli
          image: registry.redhat.io/rhoai/rhai-cli-rhel9:v3.3.5
          command: ["sh","-c","sleep infinity"]
          volumeMounts:
            - { name: backup, mountPath: /tmp/rhoai-upgrade-backup }
      volumes:
        - name: backup
          persistentVolumeClaim: { claimName: rhai-cli-backup }
EOF
----

Grant the tool's service account `cluster-admin`. Read access is enough for the assessment in this module, but the same pod runs helper scripts in Modules 3, 4, and 7 that create, patch, and delete InferenceServices, ServingRuntimes, ConfigMaps, and other cluster-scoped resources -those calls use the pod's service account token and need write:

[.console-input]
[source,role="execute"]
----
oc create clusterrolebinding rhai-cli-admin \
  --clusterrole=cluster-admin \
  --serviceaccount=rhai-migration:default
----

rhai-cli-0   1/1     Running   0          30s

=== Reading the results

The `IMPACT` column is what matters. rhai-cli emits four levels (`prohibited`, `critical`, `warning`, `info`); on this cluster the report has `critical`, `warning`, and `info`:

* *`critical`* (`✗`) - the upgrade will not proceed, or will break the cluster, until you resolve these. They come in two shapes:
** *Components 3.x removes* -*KServe Serverless*, *Service Mesh v2*, *ModelMesh Serving*, *CodeFlare*, *Kueue*. You disable each in Module 4 after you've migrated its workloads in Module 3.
** *Workload findings* you must act on -*Serverless* / *ModelMesh* InferenceServices, the *ModelMesh ServingRuntime*, and the *LlamaStackDistribution*. You convert / delete these in Module 3 (model serving) and Module 4 (LSD).
* *`warning`* (`⚠`) - things the migration handles automatically *or* that need action but won't block the upgrade by themselves. Examples here: the `datasciencepipelines` → `aipipelines` schema-rename warning (forward-looking — flagged here by the lint but not actually shipped yet in {target-version}; see Module 7 Exercise 7.4), the `inferenceservice-config` ConfigMap patch (you'll run a helper in Module 4), the 4 running workbenches that must be stopped before the upgrade (Module 5), the 2 RayClusters needing pre-upgrade backup (Module 4), the GuardrailsOrchestrator otelExporter migration (Module 7).
* *`info`* (`✓`) - the check passed; nothing to do.

[IMPORTANT]
====
Note the dependency: the `serverless-removal` and `modelmeshserving/removal` *component* criticals can only be cleared safely *after* you migrate the *workload* criticals (Serverless + ModelMesh InferenceServices and the ModelMesh ServingRuntime). Order matters -that's Module 3.
====

[.console-input]
[source,role="execute"]

oc exec -n rhai-migration rhai-cli-0 -- \

----

✓ You can see the nine critical (`✗`) findings and the ten warning (`⚠`) findings

== Exercise 2.4: The assessment is not a workload inventory

A clean or short assessment does not mean an empty cluster. The lint reports *removed components* and *incompatible* workloads -it does not list workloads that are already compatible. A RawDeployment model, for example, won't appear at all. Always inventory what's actually running before you plan the window.

Inventory the model-serving workloads directly, and note their deployment modes:

[.console-input]
[source,role="execute"]
----
oc get isvc -A -o custom-columns='NS:.metadata.namespace,NAME:.metadata.name,MODE:.metadata.annotations.serving\.kserve\.io/deploymentMode,READY:.status.conditions[?(@.type=="Ready")].status'
----

[.console-output]
----
NS               NAME                 MODE            READY
ml-project-a     my-serverless-isvc   Serverless      True
ml-project-b     another-sl-model     Serverless      True
ml-project-c     my-modelmesh-isvc    ModelMesh       True
ml-project-raw   tinyllama-1b-cpu     RawDeployment   True
----

Four real, running models -and only the first three need to move before the upgrade.

* `ml-project-a/my-serverless-isvc` and `ml-project-b/another-sl-model` are LLMs (tinyllama on vLLM) running on KServe *Serverless* mode -convert each to RawDeployment in Module 3.
* `ml-project-c/my-modelmesh-isvc` is a MobileNetV2 image classifier (ONNX) served on *ModelMesh* through OpenVINO Model Server -re-host on a single-model KServe RawDeployment runtime in Module 3.
* `ml-project-raw/tinyllama-1b-cpu` is already on KServe *RawDeployment* -3.x-compatible as-is, nothing to do.

This is also the lesson the lint already taught: it flagged the first three as impacted workloads and said nothing about `tinyllama-1b-cpu`, because RawDeployment is the supported 3.x mode. A clean lint for an ISVC means "nothing to convert here," not "no ISVC exists here."

Inventory the multi-model ServingRuntimes (the ModelMesh signal is `spec.multiModel: true`):

[.console-input]
[source,role="execute"]
----
oc get servingruntime -A -o json | jq -r '.items[] | select(.spec.multiModel==true) | "\(.metadata.namespace)/\(.metadata.name)"'
----

[.console-output]
----
ml-project-c/ovms-mm
----

LlamaStackDistribution is a *separate CRD* from `InferenceService` -it doesn't appear in the `oc get isvc` table above and it's the workload driving the `llamastackdistribution / config` blocker from Exercise 2.3:

[.console-input]
[source,role="execute"]
----

----

[.console-output]

* `ldap-user17-rag-225/lsd-llama-milvus-2323` is a Llama Stack Distribution. 3.x moves Llama Stack from SQLite to PostgreSQL with no in-place data migration path. You'll delete this LSD in Module 4 (data archival is out of scope here) and deploy a fresh 3.x-shaped LSD in Module 7.

Each of these has a 2→3 migration concern even though most don't show up as rhai-cli blockers:

== Module summary

The cluster has nine critical findings (component removals + workload conversions), ten warnings, and live workloads behind nearly every blocker. You cannot clear the Serverless and ModelMesh component criticals until you migrate the models off them. In the next module you'll do exactly that: convert the Serverless ISVCs to RawDeployment and migrate the ModelMesh model to KServe. Module 4 then runs the rest of the platform prep -backups, ConfigMap update, LSD delete, and the operator uninstalls.
