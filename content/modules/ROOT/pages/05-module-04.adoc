= Module 4: Remediate the platform
:source-highlighter: rouge
:toc: macro
:toclevels: 2

With every model converted to RawDeployment, it's safe to start the platform work -but in the *right order*. Real clusters carry workloads the workshop has so far ignored: Ray clusters, TrustyAI services, AI Pipelines. Their data must be captured *before* you flip the components that host them, because some of those flips destroy state. Llama Stack is the one exception: its 2→3 transition has no compatible upgrade path, so the action is delete pre-upgrade and recreate fresh post-upgrade (data archival is out of scope here). This module starts with backups, deletes the LSDs, then runs the disables, then the 3.x prerequisites, then operator uninstalls.

In this module you'll back up every workload whose data can be lost across the upgrade (Ray clusters, TrustyAI metrics, AI Pipelines), delete any LSDs so they can be recreated fresh in Module 7, update the `inferenceservice-config` ConfigMap with the hardware-profile ignorelist, flip the removed DSC components to `Removed`, tear down the Service Mesh v2 dependency, install Red Hat Connectivity Link (RHCL) as the 3.x llm-d auth plane, freeze the LLMInferenceService templates, and finally uninstall the 2.x serving operators.

Order matters throughout. Back up RayClusters *before* disabling Ray. The `inferenceservice-config` ConfigMap must be updated *after* every ISVC is RawDeployment but *before* the disables, so the upgrade reconciler honours the new ignorelist. KServe Serverless must drain before you remove the Service Mesh v2 dependency. The Service Mesh member roll has to be deleted before its operator can be uninstalled. And RHCL must be up *before* you uninstall the standalone Authorino, or LLMInferenceService loses auth for the duration of the window.

== Learning objectives

By the end of this module, you'll be able to:

* Back up RayClusters, TrustyAI metrics + data, and AI Pipelines DSPAs ahead of destructive component changes, and delete any LlamaStackDistributions
* Update the `inferenceservice-config` ConfigMap with the hardware-profile ignorelist
* Flip the removed DSC components (`kueue`, `ray`, `codeflare`, `modelmeshserving`, `kserve.serving`) to `Removed`
* Remove the Service Mesh v2 dependency including the lingering member roll
* Install Red Hat Connectivity Link with TLS-enabled Authorino as the llm-d auth plane
* Freeze the LLMInferenceService scheduler/router templates against the 3.x rewrite
* Uninstall the OpenShift Serverless, Service Mesh 2, and standalone Authorino operators

== Exercise 4.2: Back up TrustyAI metrics and data

TrustyAI's storage schema changed between 2.x and 3.x. Historical bias-detection metrics and training data can become unreadable without a pre-upgrade snapshot. Capture both before anything else touches the component.

Confirm TrustyAI is `Managed` (skip the rest of this exercise if not):

[.console-input]
[source,role="execute"]
----
oc get dsc default-dsc -o jsonpath='trustyai={.spec.components.trustyai.managementState}{"\n"}'
oc get trustyaiservice -A -o custom-columns='NS:.metadata.namespace,NAME:.metadata.name,STORAGE:.spec.storage.format'
----

Per the migration guide, metrics are backed up with a per-namespace port-forward + `curl` (no helper script for metrics), and data storage is backed up with the official `backup-data.sh` helper (auto-detects PVC vs DATABASE). Run both for each namespace that has a TrustyAIService.

Run the metrics + data backup for *both* TrustyAI service namespaces (`test-trustyaiservice` and `test-trustyaiservice-db`). The block below sets `BACKUP_DIR` explicitly -the rhai-cli image doesn't always set it for non-login shells, and an unset `BACKUP_DIR` makes `curl -o "$BACKUP_DIR/..."` silently write to `/trustyai-metrics-*.json` at the filesystem root where the cleanup tooling won't find it.

[.console-input]
[source,role="execute"]
----
for NS in test-trustyaiservice test-trustyaiservice-db; do
  echo "=== Backing up $NS ==="
  oc exec -n rhai-migration rhai-cli-0 -it -- bash -c "
    set -e
    export BACKUP_DIR=/tmp/rhoai-upgrade-backup/trustyai
    mkdir -p \"\$BACKUP_DIR\"
    export NS=$NS
    export TAS_NAME=\$(oc get trustyaiservice -n \"\$NS\" -o jsonpath='{.items[0].metadata.name}')
    export SVC_PORT=\$(oc get svc -n \"\$NS\" \"\$TAS_NAME\" -o jsonpath='{.spec.ports[?(@.name==\"http\")].port}')
    echo \"[INFO] NS=\$NS TAS=\$TAS_NAME PORT=\$SVC_PORT\"
    # Metrics: port-forward + curl /metrics/all/requests
    oc port-forward -n \"\$NS\" \"svc/\$TAS_NAME\" 8080:\${SVC_PORT} &
    PF_PID=\$!; sleep 3
    curl -sk -H \"Authorization: Bearer \$(oc whoami -t)\" \
      http://localhost:8080/metrics/all/requests \
      -o \"\${BACKUP_DIR}/trustyai-metrics-\${NS}-\$(date +%Y%m%d-%H%M%S).json\"
    kill \$PF_PID 2>/dev/null || true
    # Data: backup-data.sh auto-detects PVC vs DATABASE per TrustyAIService
    cd /opt/rhai-upgrade-helpers/trustyai
    ./backup-data.sh --namespace \"\$NS\"
  "
done
----

[NOTE]
====
On a fresh lab `backup-data.sh` may print `[WARN] PVC appears to be empty (0 files copied)` for `test-trustyaiservice` -no bias-detection has been run, so there's no input data. The backup still completes successfully. `test-trustyaiservice-db` produces a SQL dump (the `-db` suffix marks it as database-backed).
====

Outside the pod, capture any `GuardrailsOrchestrator` `spec.otelExporter` blocks so you can restore them post-upgrade (no helper script for this either):

[.console-input]
[source,role="execute"]
----
oc get guardrailsorchestrator -A -o json \
  | jq -r '.items[] | select(.spec.otelExporter != null) | {ns: .metadata.namespace, name: .metadata.name, otelExporter: .spec.otelExporter}' \
  > /tmp/trustyai-guardrails-otel-backup-$(date +%Y%m%d%H%M).json
----

Copy the backup off the rhai-cli pod:

[.console-input]
[source,role="execute"]
----
oc cp rhai-migration/rhai-cli-0:/tmp/rhoai-upgrade-backup/trustyai ./trustyai-backup
----

=== Verify

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -- ls -la /tmp/rhoai-upgrade-backup/trustyai
oc exec -n rhai-migration rhai-cli-0 -- find /tmp/rhoai-upgrade-backup/trustyai -maxdepth 2 -type f
----

[.console-output]
----
trustyai-metrics-test-trustyaiservice-20260218-175450.json
trustyai-metrics-test-trustyaiservice-db-20260218-175501.json
trustyai-data-test-trustyaiservice-20260218-175451/data/...
trustyai-db-test-trustyaiservice-db-20260218-175502/dump.sql

✓ The backup directory lists one `trustyai-metrics-<ns>-*.json` per namespace, plus a `trustyai-data-<ns>-*/` (PVC) tree for the PVC-backed service and a `trustyai-db-<ns>-*/dump.sql` for the database-backed one

== Exercise 4.4: Delete any Llama Stack distributions

Llama Stack's 2→3 transition is "delete and replace." The schema changed (SQLite → PostgreSQL, VectorDB API removed, Inference API became OpenAI-compatible), and there's no in-place data migration -so existing LSDs cannot be carried across. Archiving the in-pod data is out of scope for this workshop; if your LSD owner cares about telemetry/agent state, that's a separate conversation. For the migration itself, the action is just *delete*.

If no LSDs are present, this exercise is a no-op. If any exist, delete them so the upgrade has a clean slate and the `llamastackdistribution / config` blocker clears:

[.console-input]

oc get llamastackdistribution -A
oc get llamastackdistribution -A -o json \
  | jq -r '.items[]? | "\(.metadata.namespace) \(.metadata.name)"' \
  | while read NS NAME; do oc delete llamastackdistribution "$NAME" -n "$NS"; done
----

You'll deploy a fresh 3.x-shaped LSD in Module 7.

== Exercise 4.5: Update the inferenceservice-config ConfigMap

With every ISVC on RawDeployment (Module 3), apply the hardware-profile ignorelist changes to the `inferenceservice-config` ConfigMap. The official `hardwareprofiles-ignorelist.sh` helper does both jobs the migration guide §2.8.8 calls for: adds `opendatahub.io/hardware-profile-name` and `opendatahub.io/hardware-profile-namespace` to the disallowed list (so the upgrade can migrate HardwareProfiles cleanly), and prevents InferenceService redeployment during the upgrade by marking the ConfigMap unmanaged.

Back up first so you can restore the 2.x state if anything goes sideways:

[.console-input]
[source,role="execute"]
----
mkdir -p /tmp/rhoai-upgrade-backup
oc get configmap inferenceservice-config -n redhat-ods-applications -o yaml \
  > /tmp/rhoai-upgrade-backup/inferenceservice-config-backup.yaml
ls -lh /tmp/rhoai-upgrade-backup/inferenceservice-config-backup.yaml
----

Then run the helper from inside the rhai-cli container:

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -- \
  /opt/rhai-upgrade-helpers/model-serving/before-upgrade/hardwareprofiles-ignorelist.sh \
  -n redhat-ods-applications
----

=== Verify

[.console-input]
[source,role="execute"]
----
oc get configmap inferenceservice-config -n redhat-ods-applications \
  -o yaml | grep "hardware" -B 2 -A 2
oc get configmap inferenceservice-config -n redhat-ods-applications \
  -o jsonpath='managed={.metadata.annotations.opendatahub\.io/managed}{"\n"}'
----

✓ The ConfigMap lists `opendatahub.io/hardware-profile-name` and `opendatahub.io/hardware-profile-namespace` on the disallowed list, and `managed=false` so the upgrade reconciler honours the patch

[NOTE]
====
A side effect of `managed=false` is that the upgrade will *not* redeploy your model-serving pods. They keep running on the same kserve runtime image they were on. If you wanted a fresh runtime image post-upgrade (for example to pick up a newer vLLM build), restart each predictor explicitly after Module 7 once the platform is healthy.
====

== Exercise 4.6: Disable Kueue

Leaving Kueue `Managed` at upgrade time is the single highest-risk mistake -its admission webhook intercepts Job creation cluster-wide, and during the operator upgrade the webhook and its CRDs fall out of sync, which can reject Job submissions for *any* workload on the cluster.

[.console-input]
[source,role="execute"]
----
oc patch $(oc get dsc -o name | head -n1) --type=merge -p '{
  "spec": { "components": { "kueue": { "managementState": "Removed" } } }
}'
----

✓ `oc patch` reports `patched` -`kueue.managementState` is now `Removed`. The `KueueReady` DSC condition flips to `False / reason=Removed`, which is expected.

== Exercise 4.11: Install Red Hat Connectivity Link for llm-d

RHOAI 3.x routes LLMInferenceService (llm-d) traffic through Kubernetes Gateway API with *Red Hat Connectivity Link* (RHCL -upstream: Kuadrant) as the consolidated auth/policy plane. RHCL replaces the standalone Authorino that came in with the Serverless stack.

Install RHCL *before* uninstalling the standalone Authorino (Exercise 4.13), or you'll drop auth for any LLMInferenceService for the duration of the window.

=== Sub-step A: Install the RHCL operator (AllNamespaces, into `openshift-operators`)

The RHCL operator must be installed *AllNamespaces*, with the Subscription in `openshift-operators`. The `kuadrant-system` namespace is created separately and used only for the Kuadrant CR + Authorino TLS resources -*not* for the Subscription.

[IMPORTANT]
====
Earlier RHOAI 2.25 → 3.3 migration documentation directed installing RHCL into `kuadrant-system` with mode "A specific namespace on the cluster". That fails on RHCL v1.3.3:

----
phase: Failed
reason: UnsupportedOperatorGroup
message: OwnNamespace InstallModeType not supported, cannot configure to watch own namespace
----

The bundled `dns-operator` and `limitador-operator` CSVs Fail with the same error. Authorino's own CSV happens to succeed because Authorino does support OwnNamespace, but it's stranded without the rest of the bundle. RHCL v1.3.3 supports AllNamespaces only -install into `openshift-operators`.
====

[.console-input]
[source,role="execute"]
----
oc create ns kuadrant-system 2>/dev/null || true

oc apply -f - <<'EOF'
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata: { name: rhcl-operator, namespace: openshift-operators }
spec:
  channel: stable
  name: rhcl-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
  installPlanApproval: Automatic
EOF
----

`openshift-operators` ships an AllNamespaces OperatorGroup by default, so no OperatorGroup needs to be applied. Wait for the RHCL CSV (and the dns / limitador / authorino dependencies it pulls in) to reach `Succeeded`:

[.console-input]
[source,role="execute"]
----
oc get csv -n openshift-operators | grep -iE "rhcl|dns-operator|limitador|authorino"
----

Re-run until you see four `Succeeded` rows. Expected end state:

[.console-output]
----
authorino-operator.v1.3.0     Authorino Operator                 1.3.0      authorino-operator.v1.2.4     Succeeded
dns-operator.v1.3.0           DNS Operator                       1.3.0      dns-operator.v1.2.0           Succeeded
limitador-operator.v1.3.0     Limitador Operator                 1.3.0      limitador-operator.v1.2.0     Succeeded
rhcl-operator.v1.3.3          Red Hat Connectivity Link          1.3.3      rhcl-operator.v1.3.2          Succeeded
----

=== Sub-step B: Create the Kuadrant CR and enable TLS on Authorino

Create the Kuadrant CR. RHCL provisions Authorino and Limitador in response:

[.console-input]
[source,role="execute"]
----
oc apply -f - <<'EOF'
apiVersion: kuadrant.io/v1beta1
kind: Kuadrant
metadata: { name: kuadrant, namespace: kuadrant-system }
EOF
----

Check Kuadrant's current state:

[.console-input]
[source,role="execute"]
----
oc get kuadrant kuadrant -n kuadrant-system -o jsonpath='{.status.conditions}' | jq
----

Annotate the Authorino service so OpenShift's built-in service signer issues the listener cert (no cert-manager involved -OCP's internal CA mints `authorino-server-cert` automatically):

[.console-input]
[source,role="execute"]
----
oc annotate svc/authorino-authorino-authorization \
  service.beta.openshift.io/serving-cert-secret-name=authorino-server-cert \
  -n kuadrant-system

sleep 2
----

Apply the Authorino CR with TLS on the **listener only** -the migration guide explicitly leaves `oidcServer.tls.enabled` *off*:

[.console-input]
[source,role="execute"]
----
oc apply -f - <<'EOF'
apiVersion: operator.authorino.kuadrant.io/v1beta1
kind: Authorino
metadata:
  name: authorino
  namespace: kuadrant-system
spec:
  replicas: 1
  clusterWide: true
  listener:
    tls:
      enabled: true
      certSecretRef:
        name: authorino-server-cert
  oidcServer:
    tls:
      enabled: false
EOF

oc wait --for=condition=ready pod -l authorino-resource=authorino -n kuadrant-system --timeout=150s
----

=== Verify

[.console-input]
[source,role="execute"]

oc get csv -n openshift-operators | grep -iE "rhcl|dns-operator|limitador|authorino"

oc get pods -n kuadrant-system -l authorino-resource=authorino

Expect: four `Succeeded` CSVs (`rhcl-operator`, `dns-operator`, `limitador-operator`, `authorino-operator`), `listener-tls=true oidc-tls=false`, one Authorino pod `Running 1/1`, the `authorino-server-cert` Secret present and minted by OpenShift's service signer.

Enumerate the LLMInferenceServices:

== Exercise 4.13: Uninstall the 2.x serving operators

OpenShift Serverless, Service Mesh 2, and standalone Authorino were dependencies of the 2.x KServe Serverless stack. With that stack removed -and RHCL up to take over auth -uninstall them.

[.console-input]
[source,role="execute"]
----
# Standalone Authorino: delete the Subscription only, do NOT delete the CSV.
# Background (see IMPORTANT block below): RHCL is also installed into
# openshift-operators, so OLM has deduped both Subscriptions to a single shared
# authorino-operator CSV. Removing the CSV would tear down RHCL Authorino too.

== Troubleshooting

*Issue*: `oc delete smmr` hangs.

*Solution*: The member roll's finalizer is processed by the Service Mesh 2 operator. Delete the SMMR *before* uninstalling the operator in Exercise 4.13. If you already uninstalled it, reinstall it briefly so the finalizer can process, then delete the SMMR.

*Issue*: Kuadrant CR is stuck at `Ready=False` with `MissingDependency` for a Gateway API provider.

*Solution*: On a clean OCP 4.20+ cluster the Cluster Ingress Operator installs and manages `servicemeshoperator3` as the Gateway API provider -confirm with `oc get csv -n openshift-operators | grep servicemeshoperator3` (expect `Succeeded`). If the CSV is missing or `Failed` on a *disconnected* cluster, you skipped the §2.8.10.2 mirror step -mirror the SMv3 image into your registry and re-check. On a connected cluster a missing SMv3 CSV usually means the Cluster Ingress Operator itself is unhealthy; investigate `openshift-ingress-operator` before doing anything else.

*Issue*: `authorino-tls-readiness` still fails after Exercise 4.11.

*Solution*: Confirm the `authorino-server-cert` Secret was minted by OpenShift's service signer: `oc get secret authorino-server-cert -n kuadrant-system`. If it's missing, the `service.beta.openshift.io/serving-cert-secret-name` annotation didn't take -re-run `oc annotate svc/authorino-authorino-authorization ...` from Sub-step B. The Authorino CR's `listener.tls.enabled` must be `true`; `oidcServer.tls.enabled` must stay `false` (the migration guide intentionally leaves it off).

*Issue*: A converted model goes `READY=False` during the Serverless teardown.

*Solution*: Check the predictor pod with `oc get pods -n <ns> -l serving.kserve.io/inferenceservice=<name>` and its logs. RawDeployment models do not depend on Knative, so the Serverless teardown itself shouldn't affect them; pod restarts later during the upgrade are expected (this is how they pick up the newer kserve and runtime images).

*Issue*: KServe Serverless teardown seems stuck.

*Solution*: It can take 5-10 minutes. Watch `oc get pods -n knative-serving` fall to zero. If it never drains, check `oc logs -n redhat-ods-applications -l app=kserve-controller-manager`.
