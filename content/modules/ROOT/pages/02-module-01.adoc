= Module 1: Test the baseline workloads
:source-highlighter: rouge
:toc: macro
:toclevels: 2

Before you change anything, capture what *works* today. A migration without a baseline is just hope: you have no way to tell whether something failed because of the upgrade or because it was already broken. In this module you'll exercise the user-facing workloads -models, workbenches, pipelines, Ray clusters -and confirm each one responds. Module 8 runs the *same* tests post-upgrade so you can compare side by side.

== Learning objectives

By the end of this module, you'll be able to:

* List the user-facing workloads on the cluster (ISVCs, workbenches, DSPAs, Ray clusters)
* Send an inference request to each ISVC and get a sensible response
* Confirm each workbench is reachable through its 2.x Route
* Confirm the DSPA serves the pipelines API and Ray clusters are `ready`

== Exercise 1.2: Inference test each model

The same probing logic Module 8 uses post-upgrade: discover the served model ID via `/v1/models` (vLLM) or `/v2/models/<name>` (OVMS), then send a representative request. ISVCs with only an `svc.cluster.local` URL (Headless service config) skip the HTTP probe -their pod state is the smoke test.

[.console-input]
[source,role="execute"]
----
for isvc in $(oc get isvc -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}{"\n"}{end}'); do
  NS="${isvc%/*}"; NAME="${isvc##*/}"
  URL=$(oc get isvc "$NAME" -n "$NS" -o jsonpath='{.status.url}')
  MODE=$(oc get isvc "$NAME" -n "$NS" -o jsonpath='{.metadata.annotations.serving\.kserve\.io/deploymentMode}')
  # ModelMesh pods don't carry the KServe inferenceservice label; the runtime pod is the workload.
  if [ "$MODE" = "ModelMesh" ]; then
    RUNTIME=$(oc get isvc "$NAME" -n "$NS" -o jsonpath='{.spec.predictor.model.runtime}')
    POD_STATE=$(oc get pods -n "$NS" -l "name=modelmesh-serving-$RUNTIME" \
      -o jsonpath='{.items[0].status.phase}' 2>/dev/null)
  else
    POD_STATE=$(oc get pods -n "$NS" -l serving.kserve.io/inferenceservice="$NAME" \
      -o jsonpath='{.items[0].status.phase}' 2>/dev/null)
  fi
  echo "--- $isvc ---"
  echo "  URL:           $URL"
  echo "  Predictor pod: ${POD_STATE:-no pod}"
  [[ "$URL" != https://* ]] && { echo "  HTTP test:     skipped (cluster-internal only)"; continue; }
  MODEL_ID=$(curl -sk --max-time 10 "${URL}/v1/models" 2>/dev/null | jq -r '.data[0].id // empty')
  if [ -n "$MODEL_ID" ]; then
    REPLY=$(curl -sk --max-time 30 "${URL}/v1/completions" \
      -H 'Content-Type: application/json' \
      -d "{\"model\":\"$MODEL_ID\",\"prompt\":\"The capital of France is\",\"max_tokens\":8,\"temperature\":0}" \
      | jq -r '.choices[0].text // "(error)"')
    echo "  vLLM model id: $MODEL_ID"
    echo "  Completion:    $REPLY"
  else
    META=$(curl -sk --max-time 10 "${URL}/v2/models/mobilenet" | jq -r '.name // empty')
    echo "  OVMS model:    ${META:-(no metadata)}"
  fi
done
----

Expect every predictor pod `Running` and the externally-reachable LLMs to return sensible completions.

== Exercise 1.3: Confirm workbenches respond

2.x workbenches are exposed through OpenShift Routes. Curl each one — a 200, 302, or 303 confirms the pod + Route are healthy:

[.console-input]
[source,role="execute"]
----
for nb in $(oc get notebook -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}{"\n"}{end}'); do
  NS="${nb%/*}"; NAME="${nb##*/}"
  HOST=$(oc get route "$NAME" -n "$NS" -o jsonpath='{.spec.host}' 2>/dev/null)
  if [ -z "$HOST" ]; then
    echo "$nb: no Route (workbench may be stopped)"
    continue
  fi
  CODE=$(curl -sk -o /dev/null -w '%{http_code}' --max-time 10 "https://${HOST}/")
  echo "$nb https://${HOST}/ -> HTTP $CODE"
done
----

== Exercise 1.4: Confirm the DSPA and Ray clusters respond

[.console-input]
[source,role="execute"]
----
# DSPA: healthz + pipelines list
NS=dspa-sample
TOKEN=$(oc whoami -t)
DSPA_HOST=$(oc get route -n "$NS" ds-pipeline-sample-pipelines -o jsonpath='{.spec.host}' 2>/dev/null)
echo "--- DSPA https://${DSPA_HOST} ---"
curl -sk -H "Authorization: Bearer $TOKEN" "https://${DSPA_HOST}/apis/v2beta1/healthz" | jq .

# Ray: cluster state
echo ""
echo "--- Ray clusters ---"
oc get raycluster -A -o custom-columns='NS:.metadata.namespace,NAME:.metadata.name,DESIRED:.spec.workerGroupSpecs[0].replicas,AVAILABLE:.status.availableWorkerReplicas,STATUS:.status.state'
----

Expect DSPA healthz to return a healthy response and both Ray clusters to show `STATUS=ready` with `AVAILABLE` matching `DESIRED`.

The cluster's user-visible surface is known-good. If anything looked broken before you touched the cluster, fix it now -or document it as already-broken so the migration doesn't take the blame. Continue to *Module 2: Assess the cluster*.
