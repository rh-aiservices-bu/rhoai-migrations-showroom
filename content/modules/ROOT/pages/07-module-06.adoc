= Module 6: Post-upgrade verification
:source-highlighter: rouge
:toc: macro
:toclevels: 2

The operator reports {target-version} and the DSC is `Ready`, but the migration isn't finished until you've verified *every* component that was running before the upgrade is healthy on the new platform. The models matter most -but Ray, Feast, KFTO, Model Registry, AI Pipelines, Llama Stack, and TrustyAI each have their own post-upgrade tasks too.

In this module you'll confirm platform health, verify the four converted models survived, check the schema and HardwareProfile migrations, patch the workbenches onto the new auth model, run the Ray cluster migration script, verify KFTO / Feast / Model Registry / AI Pipelines / Llama Stack / TrustyAI each survived (and restore TrustyAI data if any was lost), and confirm the Gateway API is serving traffic at the new URLs.

== Learning objectives

By the end of this module, you'll be able to:

* Verify the operator, DSC, and DSCI are healthy on {target-version}
* Confirm the converted models are still serving and capture their new endpoint URLs
* Confirm the `datasciencepipelines` to `aipipelines` rename and the HardwareProfile migration
* Patch stopped workbenches onto the 3.x kube-rbac-proxy auth model
* Run the RayCluster migration script to take each cluster over to KubeRay-only management
* Verify KFTO PyTorchJobs, Feast Feature Stores, Model Registry pods, and DSPAs all survived
* Recreate Llama Stack distributions against the new 3.x schema
* Compare TrustyAI metrics + data against the pre-upgrade backups, restore lost data, check Guardrails, and break GPU deadlocks if they appear

== Exercise 6.2: Confirm the converted models still serve

This is the payoff for the conversions in Module 2. All four models should be `Ready` on RawDeployment. Their predictor pods *were* restarted during the upgrade -that's how the 3.x kserve controller rolls them onto the newer kserve runtime images (including a newer vLLM). What matters is that every ISVC recovers to `Ready=True` on the new platform:

[.console-input]
[source,role="execute"]
----
oc get isvc -A -o custom-columns='NS:.metadata.namespace,NAME:.metadata.name,MODE:.status.deploymentMode,READY:.status.conditions[?(@.type=="Ready")].status,URL:.status.url'
----

[.console-output]
----
NS               NAME                 MODE            READY   URL
ml-project-a     my-serverless-isvc   RawDeployment   True    https://my-serverless-isvc-ml-project-a...
ml-project-b     another-sl-model     RawDeployment   True    https://another-sl-model-ml-project-b...
ml-project-c     mobilenet-onnx       RawDeployment   True    https://mobilenet-onnx-ml-project-c...
ml-project-raw   tinyllama-1b-cpu     RawDeployment   True    https://tinyllama-1b-cpu-ml-project-raw...
----

Note the `URL` column -those endpoints now route through Gateway API instead of the old Routes. Two things changed during the upgrade: the `.status.url` (Routes → Gateway API hostname) and the predictor pods themselves (rolled by the 3.x kserve controller onto the newer runtime images). Capture the new URLs so you can hand them to downstream consumers:

[.console-input]
[source,role="execute"]
----
oc get isvc -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name} -> {.status.url}{"\n"}{end}'
----

[NOTE]
====
In-cluster callers using the Kubernetes Service DNS name (`<model>.<namespace>.svc.cluster.local`) are unaffected. Only *external* callers using the model's `.status.url` need the new value -the old Route-based hostname no longer resolves.
====

=== Verify

[.console-input]
[source,role="execute"]
----
oc get isvc -A -o json | jq -r '[.items[] | select(.status.conditions[]? | select(.type=="Ready" and .status=="True"))] | length'
----

[.console-output]
----
4
----

✓ All four models report `Ready=True` on RawDeployment

== Exercise 6.11: Recreate the Llama Stack distribution

You deleted any 2.x LSDs in Exercise 3.4 because the schema is incompatible (SQLite → PostgreSQL, VectorDB API removed, Inference API became OpenAI-compatible). If your cluster needs Llama Stack, deploy a fresh LSD CR matching the 3.3.3 schema now.

Key shape differences:

[cols="1,1,1"]
|===
| Field | 2.25 | 3.3.3

| Database | SQLite (in-pod) | PostgreSQL 14+ (external)
| Embedding | Implicit | *Must* declare explicit provider (e.g. `sentence-transformers`)
| Vector | VectorDB (deprecated) | Vector_IO
| Config file | `run.yaml` | `config.yaml`
| Client | `llama-stack-client` 0.2.x | `llama-stack-client` 0.4.x
|===

Follow the *Deploying a Llama Stack server* docs for 3.3.3 (link in the conclusion) for the CR shape. Recreate any agents / RAG pipelines that depended on the old LSD -they cannot reuse 2.x state.

For the lab, the original LSD was a Tech Preview workload with no production data; skip the recreation and document "not recreated" in the migration report. On a production cluster, deploy one fresh LSD per namespace that needs it.

== Troubleshooting

*Issue*: A converted model is `Ready=False` after the upgrade.

*Solution*: Check the predictor pod with `oc get pods -n <ns> -l serving.kserve.io/inferenceservice=<name>` and its logs. The 3.x kserve controller rolls each predictor onto the newer runtime image during the upgrade; a model that doesn't come back is almost always a runtime-image / model-spec mismatch (incompatible vLLM flag, missing GPU resource, model artifact path drift). The events on the predictor Deployment and the pod logs are where the cause shows up.

*Issue*: GatewayConfig stuck `Not Ready` and the dashboard URL doesn't resolve.

*Solution*: On clusters with an SRE-managed admission webhook restricting NetworkPolicy creation in `openshift-*` namespaces, the GatewayConfig reconciler's NetworkPolicy POST is rejected. Disable the operator-managed NetworkPolicy: `oc patch gatewayconfig default-gateway --type=merge -p '{"spec":{"networkPolicy":{"ingress":{"enabled":false}}}}'`.

*Issue*: The workbench helper reports `Failed: N`.

*Solution*: Do not force-start those workbenches. Inspect each with `oc describe notebook <name> -n <ns>`. The most common cause is a custom workbench image that still embeds oauth-proxy and must be rebuilt for kube-rbac-proxy.
