Congratulations -you've migrated a Red Hat OpenShift AI {source-version} cluster to {target-version} end to end.

== What you accomplished

You ran the full assessment-driven migration workflow on a live cluster:

* *Assessed* the cluster with `rhai-cli lint`, learning to separate blocking findings from advisory ones, and confirmed the assessment is not a substitute for a direct workload inventory.
* *Remediated* every blocker in the correct order: disabled the removed components (Kueue, Ray/CodeFlare, ModelMesh), patched the model-serving ConfigMap, stopped the workbenches, tore down the KServe Serverless stack, removed the Service Mesh dependency including the lingering member roll, and uninstalled the 2.x serving operators.
* *Upgraded* by switching to the `{migration-channel}` channel, approving the OLM upgrade as it walked to {target-version}, then moving to `{stable-channel}` for ongoing patches.
* *Verified* the result: DSC/DSCI healthy, the `aipipelines` schema rename, the HardwareProfile migration, workbenches patched onto kube-rbac-proxy, and Gateway API serving traffic.

== Key takeaways

* *This is a migration, not an upgrade.* Components are removed, routing and auth models change, and there is no automated rollback. The assessment-then-remediate-then-upgrade loop is how you de-risk it.
* *Order matters in remediation.* KServe Serverless must drain before the Service Mesh dependency is removed, the member roll must be deleted before its operator is uninstalled, and Kueue must be `Removed` before the upgrade to avoid cluster-wide Job-admission failures.
* *The assessment is not a workload inventory.* A clean lint means no incompatible workloads, not an empty cluster -always inventory workbenches and models directly.
* *URLs change.* The dashboard and model endpoints move from OpenShift Routes to Gateway API. Capture the old and new URLs so you can notify downstream consumers.

== What this workshop did not cover

In a production migration you would also:

* *Take a verified backup first* (etcd snapshot plus OADP) -the only rollback path for an in-place migration.
* *Plan the maintenance window* around stopping workbenches and any brief model-endpoint gap during the routing transition.
* *Notify downstream consumers* of the new model endpoint URLs before the window.
