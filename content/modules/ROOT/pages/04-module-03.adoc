= Module 3: Convert the model-serving workloads
:source-highlighter: rouge
:toc: macro
:toclevels: 2

This is the heart of the migration. Your cluster runs four models: two on KServe Serverless mode, one on ModelMesh, and one already on KServe RawDeployment. The first three are the work -both Serverless and ModelMesh are removed in 3.x, and if you disable those components while the models still depend on them, the models stop serving. The RawDeployment model is already 3.x-compatible and you leave it alone.

So you convert *first*. In this module you'll convert the two Serverless InferenceServices to RawDeployment using the backup-and-recreate pattern the admission webhook forces, then migrate the ModelMesh model to a single-model KServe RawDeployment. Only after this is done -in Module 4 -will you disable the components.

== Learning objectives

By the end of this module, you'll be able to:

* Enumerate InferenceServices by deployment mode
* Convert a Serverless ISVC to RawDeployment with backup-and-recreate (the webhook blocks in-place edits)
* Migrate a ModelMesh model to a single-model KServe RawDeployment ISVC
* Verify every model is RawDeployment and `Ready` before touching the components

== Exercise 3.1: Enumerate what needs converting

List every InferenceService that is *not* already RawDeployment:

[.console-input]

----
oc get isvc -A -o json | jq -r '.items[]
  | select((.metadata.annotations."serving.kserve.io/deploymentMode" // "") != "RawDeployment")
  | "\(.metadata.namespace)\t\(.metadata.name)\t\(.metadata.annotations."serving.kserve.io/deploymentMode")"'
----

[.console-output]
----
ml-project-a   my-serverless-isvc   Serverless
ml-project-b   another-sl-model     Serverless
ml-project-c   my-modelmesh-isvc    ModelMesh
----

Two Serverless and one ModelMesh -all three must be migrated. The already-RawDeployment ISVC (`tinyllama-1b-cpu` in `ml-project-raw`) is filtered out and needs no work.

== Exercise 3.2: Convert the first Serverless ISVC

The migration guide ships `serverless-to-raw.sh` as the official conversion helper. It's interactive: it checks prerequisites, discovers Serverless InferenceServices in the namespace, prompts you to select which to convert, and handles authentication resources and storage credentials automatically. The KServe admission webhook rejects in-place `oc patch` of `deploymentMode` (manual recreate is the documented fallback in https://access.redhat.com/articles/7134025).

Preview the conversion with `--dry-run` first:

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -it -- \
  /opt/rhai-upgrade-helpers/model-serving/before-upgrade/serverless-to-raw.sh \
  --dry-run -n ml-project-a
----

The script discovers the namespace's Serverless ISVCs and stops at interactive prompts. In `--dry-run` mode there are *two*; with `--delete-existing` (the destructive apply below) there is a *third* confirmation prompt. Step through the first two:

[.console-output]
----
📦 Available InferenceServices (Serverless deployment mode only):
[1] my-serverless-isvc (Mode: Serverless)

🤔 Please select which InferenceServices to migrate:
Enter 'all' to migrate all InferenceServices
Enter specific numbers separated by spaces (e.g., '1 3 5')
Enter 'q' to quit
Your selection:
----

Type `all` and press Enter. (For the lab there's only one ISVC, so `1` works too.)

[.console-output]
----
🏷️  Resource Naming Options:
1) Use original names (for in-place replacement)
2) Use -raw suffix (for side-by-side deployment)
Enter your choice (1 or 2):
----

Type `1` and press Enter. *In-place replacement* keeps the ISVC name the same (`my-serverless-isvc`) -that's what Module 1's baseline and Module 8's re-test query by name. The `-raw` suffix path renames everything and would break the symmetry.

In dry-run mode the script writes the generated YAML under `/tmp/rhoai-upgrade-backup/model-serving/serverless-to-raw/<isvc>/` inside the pod (an `original/` snapshot and a `raw-original-names/` rewrite). Review them, then run the same command without `--dry-run` and with `--delete-existing` to let the script handle deletion of the legacy ISVC, ServingRuntime, ServiceAccount, RoleBinding, and Istio Route in one shot:

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -it -- \
  /opt/rhai-upgrade-helpers/model-serving/before-upgrade/serverless-to-raw.sh \
  --delete-existing -n ml-project-a
----

Answer the same first two prompts (`all`, then `1`). `--delete-existing` then adds a *third* prompt:

[.console-output]
----
⚠️  WARNING: IN-PLACE REPLACEMENT MODE
Please enter 'yes' or 'no':
----

Type `yes` and press Enter. The script deletes the legacy resources and applies the RawDeployment-shaped replacements with the original names. (Anything other than `yes` here aborts.)

=== Verify

== Exercise 3.4: Understand the ModelMesh model

ModelMesh is *multi-model serving* -one ServingRuntime hosts many models, keyed by a storage path. 3.x has no ModelMesh, so the model must be re-deployed as a *single-model* KServe RawDeployment ISVC backed by a single-model ServingRuntime.

Inspect the ModelMesh ISVC and its multi-model runtime:

[.console-input]
[source,role="execute"]
----
oc get isvc my-modelmesh-isvc -n ml-project-c -o yaml | yq '.spec'
echo "--- multi-model ServingRuntime ---"
oc get servingruntime ovms-mm -n ml-project-c -o jsonpath='multiModel={.spec.multiModel}{"\n"}'
----

[.console-output]
----
predictor:
  model:
    modelFormat:
      name: onnx
      version: "1"
    runtime: ovms-mm
    storage:
      key: pvc-models
      path: mobilenet
--- multi-model ServingRuntime ---
multiModel=true
----

This is a MobileNetV2 image classifier in *ONNX* format, served by OpenVINO Model Server (OVMS) in ModelMesh multi-model mode. The model lives on a PVC (`model-store`) under the OVMS layout `mobilenet/1/`. To migrate it you re-deploy the *same model* on a *single-model* OVMS ServingRuntime as a KServe RawDeployment ISVC, pointing `storageUri` at the same PVC path. ONNX is a first-class format for both ModelMesh OVMS and single-model KServe OVMS, so this is a true re-host -no model conversion, no data movement.

Confirm the model file is present on the PVC:

[.console-input]
[source,role="execute"]
----
oc -n ml-project-c get pvc model-store -o jsonpath='PVC: {.status.phase}{"\n"}'
----

=== Verify

[.console-input]
[source,role="execute"]
----
oc get servingruntime ovms-mm -n ml-project-c -o jsonpath='{.spec.multiModel}{"\n"}'
----

✓ Output is `true` -confirming this is a ModelMesh multi-model runtime

== Exercise 3.5: Re-host the model on KServe RawDeployment

The migration guide ships `modelmesh-to-raw.sh` (§2.8.7.2) as the official ModelMesh re-host helper. It's interactive: it discovers ModelMesh ISVCs in the source namespace, prompts you to select models, choose a runtime template (single-model OVMS for ONNX, vLLM for LLMs, etc.), configures storage, and creates the new RawDeployment ServingRuntime + InferenceService in a target namespace.

[NOTE]
====
The model-store PVC is `ReadWriteOnce` (the default on `gp3-csi`). RWO means only one node can attach the volume at a time. Before the helper applies the new ISVC, release the PVC by scaling the ModelMesh runtime to zero — otherwise the new pod can land on a different node and hang with `Multi-Attach error`. On RWX storage (EFS, CephFS, NFS) this scale-down is not required.
====

Release the PVC first:

[.console-input]
[source,role="execute"]
----
oc -n ml-project-c patch servingruntime ovms-mm --type=merge -p '{"spec":{"replicas":0}}'
oc -n ml-project-c wait --for=delete pod -l name=modelmesh-serving-ovms-mm --timeout=120s
----

For the lab, the new RawDeployment ISVC lives in the *same* namespace as the ModelMesh original (`ml-project-c`). The `modelmesh-to-raw.sh` script has two flag-combinations:

* `--from-ns <A> --target-ns <B>` -cross-namespace, supports `--dry-run` for a safe preview. Source and target must differ.
* `--from-ns <A> --preserve-namespace` -*in-place*, destructive. Cannot be combined with `--dry-run` (the script errors out: `✗ Error: --dry-run and --preserve-namespace cannot be used together`).

[NOTE]
====
For in-place migration there's no dry-run preview — the script is destructive by design. The interactive prompts are the safety net: it asks for a destructive confirmation up front, then which ISVCs to migrate, which runtime template, and what to do with the storage config *before* applying anything, so you can `q` out (or answer `no`) if the choices look wrong. If you want to inspect the generated YAML before any cluster changes, dry-run against a throwaway target namespace first (see *Optional preview* below).
====

Run the in-place migration:

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -it -- \
  /opt/rhai-upgrade-helpers/model-serving/before-upgrade/modelmesh-to-raw.sh \
  --from-ns ml-project-c --preserve-namespace
----

Step through the four interactive prompts:

1. *Destructive-mode confirmation* — the script prints a `⚠️ DESTRUCTIVE OPERATION WARNING` banner and asks `Do you understand the risks and want to continue? (type 'yes' to proceed):`. Type `yes`. Anything else aborts with `👋 Migration cancelled for safety.`
2. *Selection prompt* — answer `all` to select every ModelMesh ISVC.
3. *Runtime-template prompt* — the script lists every ServingRuntime template available in the cluster. Press `d` to accept the default, `kserve-ovms` (KServe single-model OpenVINO Model Server). Don't pick `[6] ovms` -that's the multi-model variant you're migrating away from. `kserve-ovms` reads the same `mobilenet/1/model.onnx` from the same PVC, so no model conversion happens.
4. *Storage-configuration prompt* — the script detects the OpenVINO-style storage and asks whether to update it. Type `1` to *keep current configuration*. (The script transcribes the ModelMesh storage block as-is; you'll fix the resulting webhook error with a small patch after Verify below.)

[NOTE]
====
The default-runtime choice depends on the model format:

* *ONNX or OpenVINO IR* (this lab) → `kserve-ovms`
* *LLM via vLLM (CPU)* → `vllm-cpu-runtime-template`
* *LLM via vLLM (NVIDIA GPU)* → `vllm-cuda-runtime-template`
* *Caikit-wrapped LLM* → `caikit-tgis-serving-template`

If the script's default matches your model's format, accept it with `d`. If not, type the corresponding number.
====

`--preserve-namespace` keeps the original name. The ISVC stays `my-modelmesh-isvc` -only `.status.deploymentMode` flips from `ModelMesh` to `RawDeployment`. Exercise 3.6 cleans up the legacy multi-model ServingRuntime.

=== Verify

Re-run until `READY=True`:

[.console-input]
[source,role="execute"]
----
oc get isvc my-modelmesh-isvc -n ml-project-c -o jsonpath='mode={.status.deploymentMode} ready={.status.conditions[?(@.type=="Ready")].status}{"\n"}'
----

[.console-output]
----
mode=RawDeployment ready=True
----

[IMPORTANT]
====
*Known `modelmesh-to-raw.sh` bug — PVC storage block isn't translated.* The script transcribes the original ModelMesh storage block (`storage: { key: pvc-models, path: mobilenet }`) verbatim into the new ISVC. The KServe RawDeployment admission webhook rejects this with:

----
Error creating: admission webhook "inferenceservice.kserve-webhook-server.pod-mutator" denied the request:
storage type must be one of [s3, hdfs, webhdfs]. storage type [pvc] is not supported
----

If you see this in `oc get events -n ml-project-c`, patch the ISVC to use the KServe `storageUri` form instead:

[source,bash]
----
oc patch isvc my-modelmesh-isvc -n ml-project-c --type=json -p='[
  {"op":"remove","path":"/spec/predictor/model/storage"},
  {"op":"add","path":"/spec/predictor/model/storageUri","value":"pvc://model-store/mobilenet"}
]'
----

The predictor pod should come up within a minute. `pvc://model-store/mobilenet` points at the OVMS-layout root; OVMS auto-discovers the `1/` version subdirectory.
====

== Exercise 3.6: Remove the ModelMesh ISVC and runtime

Sweep for leftover ModelMesh resources. After in-place `--preserve-namespace` both selectors typically return zero rows (the ISVC was converted in place; the script already removed the multi-model ServingRuntime). For cross-namespace or `-raw`-suffix migrations they catch the leftovers:

[.console-input]
[source,role="execute"]
----
oc get isvc -n ml-project-c -o json | jq -r '.items[]
  | select(.status.deploymentMode == "ModelMesh"
        or .metadata.annotations["serving.kserve.io/deploymentMode"] == "ModelMesh")

  | select(.spec.multiModel==true)
  | .metadata.name' \
  | while read -r name; do echo "Deleting multi-model ServingRuntime: $name"; oc delete servingruntime "$name" -n ml-project-c; done
----

== Troubleshooting

*Issue*: `oc apply` of the recreated ISVC fails with a `deploymentMode` webhook error.

*Solution*: This shouldn't happen when you use the helper script — `serverless-to-raw.sh` creates the new RawDeployment ISVC as a fresh object, which the webhook accepts. The webhook only blocks *changing* mode on an existing object. If you took the manual fallback path (https://access.redhat.com/articles/7134025), confirm you flipped the annotation in your backup file and deleted the old ISVC before re-applying.

*Issue*: The recreated ISVC sits `READY=False` for several minutes.

*Solution*: RawDeployment pulls and loads the model fresh. Watch the predictor pod: `oc get pods -n <ns> -l serving.kserve.io/inferenceservice=<name>`. A CPU model can take a few minutes. Check pod logs if it never becomes ready.

*Issue*: You can't delete the multi-model ServingRuntime -"still in use".

*Solution*: Delete the ModelMesh ISVC first (Exercise 3.5). The runtime can only be removed once no ISVC references it.

*Issue*: `oc delete isvc` on a leftover Serverless ISVC hangs forever -it stays in `Terminating` and the delete never returns.

*Solution*: This is a finalizer deadlock, and it's exactly why this module converts (and deletes) every Serverless ISVC *before* Module 4 removes Serverless and Service Mesh. A Serverless ISVC carries a KServe finalizer whose cleanup deletes the Knative `Service` and Istio `VirtualService` it owns. If those components are already gone, the finalizer can't reach their APIs, errors out, and never clears -so the object is stuck. Recovery: if the operators are still installed, re-enable Serverless/Service Mesh, let the delete finish, then remove them in the right order. If they're already uninstalled, force-clear the finalizer and recreate the model as RawDeployment: `oc patch isvc <name> -n <ns> --type=merge -p '{"metadata":{"finalizers":null}}'` (this skips the real cleanup, so sweep for orphaned `ksvc`/`virtualservice` afterward). The rule: never disable a component while a workload still depends on it.
