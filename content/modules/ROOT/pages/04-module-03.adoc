= Module 3: Remediate the platform
:source-highlighter: rouge
:toc: macro
:toclevels: 2

With every model converted to RawDeployment, it's safe to start the platform work -but in the *right order*. Real clusters carry workloads the workshop has so far ignored: Ray clusters, TrustyAI services, AI Pipelines. Their data must be captured *before* you flip the components that host them, because some of those flips destroy state. Llama Stack is the one exception: its 2→3 transition has no compatible upgrade path, so the action is delete pre-upgrade and recreate fresh post-upgrade (data archival is out of scope here). This module starts with backups, deletes the LSDs, then runs the disables, then the 3.x prerequisites, then operator uninstalls.

In this module you'll back up every workload whose data can be lost across the upgrade (Ray clusters, TrustyAI metrics, AI Pipelines), delete any LSDs so they can be recreated fresh in Module 6, flip the removed DSC components to `Removed`, tear down the Service Mesh v2 dependency, install Red Hat Connectivity Link (RHCL) as the 3.x llm-d auth plane, freeze the LLMInferenceService templates, and finally uninstall the 2.x serving operators.

The model-serving pods *will* restart during the upgrade -that's deliberate. Letting the 3.x kserve controller reconcile them picks up the newer kserve and runtime images (including a newer vLLM) without a separate maintenance window. If your environment cannot tolerate a brief restart, the `inferenceservice-config` ConfigMap can be marked unmanaged before the upgrade to suppress this; that path isn't covered here.

Order matters throughout. Back up RayClusters *before* disabling Ray. KServe Serverless must drain before you remove the Service Mesh v2 dependency. The Service Mesh member roll has to be deleted before its operator can be uninstalled. And RHCL must be up *before* you uninstall the standalone Authorino, or LLMInferenceService loses auth for the duration of the window.

== Learning objectives

By the end of this module, you'll be able to:

* Back up RayClusters, TrustyAI metrics + data, and AI Pipelines DSPAs ahead of destructive component changes, and delete any LlamaStackDistributions
* Flip the removed DSC components (`kueue`, `ray`, `codeflare`, `modelmeshserving`, `kserve.serving`) to `Removed`
* Remove the Service Mesh v2 dependency including the lingering member roll
* Install Red Hat Connectivity Link with TLS-enabled Authorino as the llm-d auth plane
* Freeze the LLMInferenceService scheduler/router templates against the 3.x rewrite
* Uninstall the OpenShift Serverless, Service Mesh 2, and standalone Authorino operators

== Exercise 3.3: AI Pipelines pre-upgrade check

DSPA pipelines themselves keep running across the upgrade, but the schema changed between 2.25 and 3.3 -deprecated fields (the old `instructLab` block) and any leftover `v1alpha1` DSPA CRs need detection. The rhai-cli helper enumerates every issue.

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -- \
  bash /opt/rhai-upgrade-helpers/ai_pipelines/check_before_upgrade.sh
----

Possible outcomes:

* **OK** → no action; the helper also writes a pre-upgrade pods snapshot to `/tmp/rhoai-upgrade-backup/ai_pipelines/` which `post_upgrade_check.sh` will diff against in Module 6.
* **Warning: deprecated `instructLab` field** → safe to ignore; the field is tolerated during upgrade.
* **Deprecated v1alpha1 DSPA found** → patch each CR to `apiVersion: …/v1` and `spec.dspVersion: v2`, then patch the CRD's `.status.storedVersions` to remove `v1alpha1`.
* **Custom RBAC roles detected (ArgoCD-managed)** → advisory only; leave as-is or update the GitOps source.

If the helper flags `v1alpha1` DSPAs, the patch loop is:

[.console-input]
[source,role="execute"]
----
oc get crd datasciencepipelinesapplications.datasciencepipelinesapplications.opendatahub.io \
  -o jsonpath='{.status.storedVersions}'; echo
# If you see ["v1alpha1","v1"] and every DSPA is now v1:
# oc patch crd datasciencepipelinesapplications.datasciencepipelinesapplications.opendatahub.io \
#   --subresource=status --type=merge -p '{"status":{"storedVersions":["v1"]}}'
----

=== Verify

[.console-input]
[source,role="execute"]
----
oc get dspa -A -o custom-columns='NS:.metadata.namespace,NAME:.metadata.name,API:.apiVersion,DSP:.spec.dspVersion'
oc exec -n rhai-migration rhai-cli-0 -- ls /tmp/rhoai-upgrade-backup/ai_pipelines/

== Exercise 3.4: Delete any Llama Stack distributions

Llama Stack's 2→3 transition is "delete and replace." The schema changed (SQLite → PostgreSQL, VectorDB API removed, Inference API became OpenAI-compatible), and there's no in-place data migration -so existing LSDs cannot be carried across. Archiving the in-pod data is out of scope for this workshop; if your LSD owner cares about telemetry/agent state, that's a separate conversation. == Exercise 3.5: Disable Kueue

RHOAI 2.25 deprecated the embedded Kueue distribution; 3.3 removes it. Leaving Kueue `Managed` at upgrade time is the highest-risk mistake in the whole migration: the Kueue admission webhook intercepts Job creation cluster-wide, and during the operator upgrade the webhook and its CRDs fall out of sync, which can reject Job submissions for *any* workload on the cluster.

[.console-input]
[source,role="execute"]
----

  "spec": { "components": { "kueue": { "managementState": "Removed" } } }
}'
----

False Removed

✓ `kueue` is `Removed`. `KueueReady=False` with reason `Removed` is expected

== Exercise 3.10: Install Red Hat Connectivity Link for llm-d

RHOAI 3.x routes LLMInferenceService (llm-d) traffic through Kubernetes Gateway API with *Red Hat Connectivity Link* (RHCL -upstream: Kuadrant) as the consolidated auth/policy plane. RHCL replaces the standalone Authorino that came in with the Serverless stack.

Install RHCL *before* uninstalling the standalone Authorino (Exercise 3.12), or you'll drop auth for any LLMInferenceService for the duration of the window.

=== Sub-step A: Install the RHCL operator

The RHCL operator installs into the `kuadrant-system` namespace with **install mode "A specific namespace on the cluster"** (single-namespace) -*not* AllNamespaces / `openshift-operators`. The OperatorHub UI creates the namespace if it doesn't exist; the equivalent `oc apply` form:

[.console-input]
[source,role="execute"]
----
oc create ns kuadrant-system 2>/dev/null || true

oc apply -f - <<'EOF'
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata: { name: kuadrant-system, namespace: kuadrant-system }
spec:
  targetNamespaces:
    - kuadrant-system
---
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata: { name: rhcl-operator, namespace: kuadrant-system }
spec:
  channel: stable
  name: rhcl-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
  installPlanApproval: Automatic
EOF
----

Wait for the RHCL CSV in `kuadrant-system` to reach `Succeeded`:

[.console-input]
[source,role="execute"]
----
oc get csv -n kuadrant-system | grep rhcl
----

Re-run until you see `Succeeded`.

=== Sub-step B: Create the Kuadrant CR and enable TLS on Authorino

Create the Kuadrant CR. RHCL provisions Authorino and Limitador in response:

[.console-input]
[source,role="execute"]
----
oc apply -f - <<'EOF'
apiVersion: kuadrant.io/v1beta1
kind: Kuadrant
metadata: { name: kuadrant, namespace: kuadrant-system }
EOF

oc wait Kuadrant -n kuadrant-system kuadrant --for=condition=Ready --timeout=10m
----

Annotate the Authorino service so OpenShift's built-in service signer issues the listener cert (no cert-manager involved -OCP's internal CA mints `authorino-server-cert` automatically):

[.console-input]
[source,role="execute"]
----
oc annotate svc/authorino-authorino-authorization \
  service.beta.openshift.io/serving-cert-secret-name=authorino-server-cert \
  -n kuadrant-system

sleep 2
----

Apply the Authorino CR with TLS on the **listener only** -the migration guide explicitly leaves `oidcServer.tls.enabled` *off*:

[.console-input]
[source,role="execute"]
----
oc apply -f - <<'EOF'
apiVersion: operator.authorino.kuadrant.io/v1beta1
kind: Authorino
metadata:
  name: authorino
  namespace: kuadrant-system
spec:
  replicas: 1
  clusterWide: true
  listener:
    tls:
      enabled: true
      certSecretRef:
        name: authorino-server-cert
  oidcServer:
    tls:
      enabled: false
EOF

oc wait --for=condition=ready pod -l authorino-resource=authorino -n kuadrant-system --timeout=150s
----

=== Verify

[.console-input]
[source,role="execute"]
----
oc get csv -n kuadrant-system | grep rhcl
oc get kuadrant kuadrant -n kuadrant-system -o jsonpath='{.status.conditions}' | jq
oc get authorino authorino -n kuadrant-system -o jsonpath='listener-tls={.spec.listener.tls.enabled}{"\n"}'
oc get pods -n kuadrant-system -l authorino-resource=authorino
----

Expect: RHCL CSV `Succeeded`, Kuadrant condition `Ready=True` with message `Kuadrant is ready`, `listener-tls=true`, Authorino pods `Running 1/1`.

✓ RHCL is installed, Kuadrant is `Ready`, Authorino has TLS on the listener -the `kuadrant-readiness` and `authorino-tls-readiness` blockers are cleared

== Exercise 3.11: Freeze the LLMInferenceService template annotations

The 3.x upgrade rewrites the llm-d scheduler/router templates in the `inferenceservice-config` ConfigMap. If an LLMInferenceService is implicitly using the 2.25 template defaults, the rewrite can leave its scheduler dropping pods after the upgrade. Pin the template names on the LLMInferenceService's `.status.annotations` to freeze the 2.25 behaviour across the upgrade.

The pins go on the `status` subresource, *not* `metadata.annotations` -that's where the 2.25 scheduler reads them.

Enumerate the LLMInferenceServices:

[.console-input]
[source,role="execute"]
----
oc get llminferenceservice -A
----

[.console-output]
----
NAMESPACE       NAME           URL   READY   REASON                       AGE
llm-project-a   my-llm-model         False   MinimumReplicasUnavailable   90m
----

Apply the eight template pins to each LLMInferenceService (`llmisvc` is the short kind name):

[.console-input]
[source,role="execute"]
----
oc patch llmisvc my-llm-model -n llm-project-a --subresource=status --type=merge -p '{
  "status": {
    "annotations": {
      "serving.kserve.io/config-llm-template":                        "kserve-config-llm-template",
      "serving.kserve.io/config-llm-decode-template":                 "kserve-config-llm-decode-template",
      "serving.kserve.io/config-llm-worker-data-parallel":            "kserve-config-llm-worker-data-parallel",
      "serving.kserve.io/config-llm-decode-worker-data-parallel":     "kserve-config-llm-decode-worker-data-parallel",
      "serving.kserve.io/config-llm-prefill-template":                "kserve-config-llm-prefill-template",
      "serving.kserve.io/config-llm-prefill-worker-data-parallel":    "kserve-config-llm-prefill-worker-data-parallel",
      "serving.kserve.io/config-llm-scheduler":                       "kserve-config-llm-scheduler",
      "serving.kserve.io/config-llm-router-route":                    "kserve-config-llm-router-route"
    }
  }
}'
----

[NOTE]
====
On this cluster `my-llm-model` is `Ready=False` -that's a model-level health issue (typically a missing GPU resource or model artifact), not a migration concern. The template freeze applies regardless of whether the workload is currently serving; it pins the scheduler's behaviour for after the upgrade.
====

=== Verify

[.console-input]
[source,role="execute"]
----
oc get llmisvc my-llm-model -n llm-project-a -o jsonpath='{.status.annotations}' | jq 'to_entries | map(select(.key | startswith("serving.kserve.io/config-llm-"))) | length'
----

[.console-output]
----
8
----

✓ All eight `serving.kserve.io/config-llm-*` annotations are pinned on the LLMInferenceService

== Exercise 3.12: Uninstall the 2.x serving operators

OpenShift Serverless, Service Mesh 2, and standalone Authorino were dependencies of the 2.x KServe Serverless stack. With that stack removed -and RHCL up to take over auth -uninstall them.

[.console-input]
[source,role="execute"]
----
# Standalone Authorino (came in with Serverless) - capture the installedCSV first so we
# only delete the CSV the standalone subscription installed, not the RHCL-bundled one
AUTH_CSV=$(oc -n openshift-operators get subscription authorino-operator -o jsonpath='{.status.installedCSV}' 2>/dev/null || true)
oc -n openshift-operators delete subscription authorino-operator --ignore-not-found
[ -n "$AUTH_CSV" ] && oc -n openshift-operators delete csv "$AUTH_CSV" --ignore-not-found

# OpenShift Serverless
oc -n openshift-serverless delete subscription serverless-operator --ignore-not-found
oc get csv -A 2>/dev/null | awk '/serverless-operator\./ {print "-n "$1, $2}' | xargs -L1 oc delete csv

# Service Mesh 2 (safe now that the SMCP and SMMR are gone)
oc -n openshift-operators delete subscription servicemeshoperator --ignore-not-found
oc get csv -A 2>/dev/null | awk '/servicemeshoperator\./ {print "-n "$1, $2}' | xargs -L1 oc delete csv
----

[IMPORTANT]
====
The Authorino cleanup is surgical -it deletes *only* the CSV that the standalone `authorino-operator` subscription installed. The Authorino CR provisioned by RHCL in `kuadrant-system` (which you set up in Exercise 3.10) is unaffected and continues to serve auth for LLMInferenceService.

== Troubleshooting

*Issue*: `oc delete smmr` hangs.

*Solution*: The member roll's finalizer is processed by the Service Mesh 2 operator. Delete the SMMR *before* uninstalling the operator in Exercise 3.12. If you already uninstalled it, reinstall it briefly so the finalizer can process, then delete the SMMR.

*Issue*: Kuadrant CR is stuck at `Ready=False` with `MissingDependency` for a Gateway API provider.

*Solution*: On a clean OCP 4.20+ cluster the Cluster Ingress Operator installs and manages `servicemeshoperator3` as the Gateway API provider -confirm with `oc get csv -n openshift-operators | grep servicemeshoperator3` (expect `Succeeded`). If the CSV is missing or `Failed` on a *disconnected* cluster, you skipped the §2.8.10.2 mirror step -mirror the SMv3 image into your registry and re-check. On a connected cluster a missing SMv3 CSV usually means the Cluster Ingress Operator itself is unhealthy; investigate `openshift-ingress-operator` before doing anything else.

*Issue*: `authorino-tls-readiness` still fails after Exercise 3.10.

*Solution*: Confirm the `authorino-server-cert` Secret was minted by OpenShift's service signer: `oc get secret authorino-server-cert -n kuadrant-system`. If it's missing, the `service.beta.openshift.io/serving-cert-secret-name` annotation didn't take -re-run `oc annotate svc/authorino-authorino-authorization ...` from Sub-step B. The Authorino CR's `listener.tls.enabled` must be `true`; `oidcServer.tls.enabled` must stay `false` (the migration guide intentionally leaves it off).

*Issue*: A converted model goes `READY=False` during the Serverless teardown.

*Solution*: Check the predictor pod with `oc get pods -n <ns> -l serving.kserve.io/inferenceservice=<name>` and its logs. RawDeployment models do not depend on Knative, so the Serverless teardown itself shouldn't affect them; pod restarts later during the upgrade are expected (this is how they pick up the newer kserve and runtime images).

*Issue*: KServe Serverless teardown seems stuck.
