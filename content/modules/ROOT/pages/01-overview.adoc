= Workshop overview: the migration you're about to run
:toc:
:toc-placement: preamble
:icons: font

== Your role

You are the *cluster administrator* responsible for a production Red Hat OpenShift AI {source-version} environment. Red Hat has released OpenShift AI 3.x, and your organization wants to move to it. You own the migration end to end -and your cluster is not a clean demo. It runs models on serving stacks that 3.x removes, and workbenches built for the old authentication model.

== Why this is a migration, not an upgrade

Previous OpenShift AI version bumps were routine operator upgrades. The move to 3.x is different. In a single transition:

* *Components are removed*: KServe Serverless mode, ModelMesh Serving, the embedded Kueue, and CodeFlare are all gone in 3.x. The embedded Service Mesh dependency is dropped. You cannot simply disable these -any workload running on them must be migrated off *first*, or it stops serving.
* *Routing changes*: OpenShift Routes are replaced by Kubernetes Gateway API. Model endpoint URLs and the dashboard URL change.
* *Authentication changes*: the oauth-proxy sidecar is replaced by kube-rbac-proxy. Workbenches built for 2.x must be patched, and custom images rebuilt.
* *Schema changes*: HardwareProfiles move to a new API group with renamed objects. (Architectural notes also describe a future DSC `datasciencepipelines` → `aipipelines` rename; that rename has not shipped in {target-version} and the DSC schema still uses `datasciencepipelines`.)

Because so much changes at once, *there is no automated rollback*. In production, the only way back is restoring from a verified backup. In this lab the cluster is disposable, so you'll focus on the migration mechanics.

== The order that matters

The single most important idea in this workshop is *sequencing*. You must migrate workloads off a component before you remove the component:

. *Convert the Serverless ISVCs to RawDeployment* -so they survive when Serverless mode is removed.
. *Migrate the ModelMesh model to KServe* -so it survives when ModelMesh is removed.
. *Only then* disable Serverless, ModelMesh, Kueue, and CodeFlare on the DSC.
. *Tear down the Service Mesh dependency*, including a member roll that the operator leaves behind.
. *Handle the workbenches* -bump images you can, plan rebuilds for the rest, and stop them all.
. *Upgrade*, then verify everything came back.

Get the order wrong -disable a component while a workload still depends on it -and you lose a running model.

This workshop is also bookended by *baseline* and *re-test* modules. Module 1 exercises every user-facing workload before you touch anything, saving the responses to disk. Module 8 re-runs the same tests against the migrated endpoints and diffs against that baseline. A migration that didn't break anything is one you can prove didn't break anything.

== The assessment-driven workflow

Red Hat ships an assessment tool, `rhai-cli`, that inspects the cluster and reports what must change. You'll run it, remediate the findings, re-run it between phases, and only upgrade once it's clean. A key lesson in Module 2: the assessment reports *incompatible* workloads and *removed* components -it is **not** a full inventory. Compatible workloads won't appear, so you'll always verify what's running directly.

== Learning objectives

By the end of this workshop you will be able to:

* Capture a pre-migration baseline of every user-facing workload and diff against it post-upgrade
* Run `rhai-cli lint` and distinguish critical findings from warnings -and know its limits as an inventory
* Convert Serverless InferenceServices to RawDeployment using the migration guide's `serverless-to-raw.sh` helper
* Migrate a ModelMesh model to a single-model KServe RawDeployment ISVC
* Disable removed components in the correct order, and tear down the Service Mesh dependency cleanly
* Discover and triage custom workbench images for the new auth and routing model
* Drive the OLM-managed upgrade to {target-version} and verify the migrated cluster end to end
