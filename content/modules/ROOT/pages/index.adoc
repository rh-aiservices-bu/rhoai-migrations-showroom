= Migrating Red Hat OpenShift AI 2.25 to 3.3
:source-highlighter: rouge
:toc: macro
:toclevels: 2

toc::[]

Welcome to the Red Hat OpenShift AI migration workshop. You'll take a *production-shaped* OpenShift AI 2.25 cluster through an in-place migration to 3.3 -and unlike a clean demo cluster, this one has real workloads that don't survive a naive upgrade.

This is a *migration*, not a routine version bump. Moving from 2.25 to 3.x changes the routing layer (OpenShift Routes to Kubernetes Gateway API), the authentication model (oauth-proxy to kube-rbac-proxy), and *removes* several components your workloads currently depend on: Serverless model serving, ModelMesh, embedded Kueue, and CodeFlare. You can't just disable those components -you have to migrate the workloads off them first, in the right order, or you lose running models.

== Prerequisites

This workshop assumes you already have:

* *OpenShift administration skills* - you can work with the `oc` CLI, read and patch custom resources, and understand operators and OLM subscriptions
* *Familiarity with Red Hat OpenShift AI* - you know what a `DataScienceCluster` (DSC) and `DSCInitialization` (DSCI) are, what an `InferenceService` (ISVC) and `ServingRuntime` are, and roughly what the components do
* *Comfort reading and editing YAML* - several steps back up, edit, and recreate custom resources

*What this workshop does NOT cover*:

* Installing OpenShift AI 2.25 from scratch (your lab cluster already has it)
* Backup and restore (a real migration requires a verified backup first; that is out of scope here)
* Rolling back a failed migration

== Log in to the cluster

Every command in this workshop runs from the terminal. Log in as `cluster-admin`:

[.console-input]
[source,role="execute",subs="attributes+"]
----
oc login -u {openshift_cluster_admin_username} -p {openshift_cluster_admin_password} {openshift_api_server_url}
----

Confirm the session has `cluster-admin` privileges:

[.console-input]
[source,role="execute"]
----
oc whoami
oc auth can-i '*' '*' --all-namespaces
----

You should see your username and `yes`. If the auth check returns `no`, log out and log back in with a cluster-admin account before continuing.

== Disable ArgoCD auto-sync

[IMPORTANT]
====
*You cannot proceed with the workshop until this step is complete.*

The lab environment is bootstrapped with OpenShift GitOps (ArgoCD). Every `Application` ships with `syncPolicy.automated.selfHeal: true`, meaning ArgoCD will *revert* any change you make on the cluster back to what its Git source says within seconds. If you skip this step, ArgoCD will undo every subscription patch, DSC edit, ISVC conversion, ConfigMap change, and operator uninstall you perform in the following modules -often before you finish reading the verify block.

Disable auto-sync on every ArgoCD Application before touching anything else. Manual sync from the ArgoCD UI still works if you ever need it, but nothing will happen unattended.
====

[.console-input]
[source,role="execute"]
----
oc get applications.argoproj.io -A -o jsonpath='{range .items[*]}{.metadata.namespace} {.metadata.name}{"\n"}{end}' | \
  while read -r NS NAME; do
    oc -n "$NS" patch application.argoproj.io "$NAME" --type=merge \
      -p='{"spec":{"syncPolicy":{"automated":null}}}'
  done
----

=== Verify

Every row must show `<none>` in the `AUTOMATED` column. If any row still shows a policy, re-run the disable loop before continuing:

[.console-input]
[source,role="execute"]
----
oc get applications.argoproj.io -A \
  -o custom-columns='NS:.metadata.namespace,NAME:.metadata.name,AUTOMATED:.spec.syncPolicy.automated'
----

== The starting cluster

Your lab cluster was provisioned with OpenShift AI {source-version} in a realistic, messy pre-migration state. The DSC is `Ready` and the cluster is healthy -but it carries workloads that the 3.x removals would break:

* *Serverless model-serving ISVCs* in `ml-project-a` and `ml-project-b` -running on KServe Serverless mode, which 3.x removes
* *A ModelMesh model* in `ml-project-c` -multi-model serving, which 3.x removes entirely
* *A custom (BYON) workbench image* registered in the dashboard -built for the 2.x auth model
* *Sample workbenches and a Data Science Pipelines application*
* The 2.x serving stack: Service Mesh 2, OpenShift Serverless, and standalone Authorino

Your job is to carry all of it cleanly onto {target-version}.

== What you'll do

[cols="1,3",options="header"]
|===
| Module | What you'll accomplish
| *Module 1: Test baseline* | Exercise every user-facing workload (inference calls, workbenches, pipelines, Ray jobs) and save a baseline snapshot so you can compare before vs after.
| *Module 2: Assess* | Run the `rhai-cli` assessment, read its findings, and learn why a clean lint is not a workload inventory.
| *Module 3: Convert serving* | Convert the Serverless ISVCs to RawDeployment and migrate the ModelMesh model to KServe -*before* the components that host them are removed.
| *Module 4: Remediate platform* | Flip the removed components to `Removed`, patch the model-serving ConfigMap, remove the Service Mesh dependency, and uninstall the 2.x operators.
| *Module 5: Workbench images* | Discover the custom workbench image, learn what a custom image needs to change for 3.x, re-register it to a rebuilt image, bump out-of-date tags, and stop every workbench.
| *Module 6: Upgrade* | Switch to the migration channel, drive the OLM upgrade to {target-version}, then move to the stable channel.
| *Module 7: Verify* | Confirm the platform is healthy, the converted models still serve, the schema and HardwareProfiles migrated, and workbenches are patched onto the new auth model.
| *Module 8: Re-test* | Re-run every test from Module 1 against the migrated endpoints and diff against the baseline.
|===
