= Module 5: Handle the workbench images
:source-highlighter: rouge
:toc: macro
:toclevels: 2

The last thing to handle before the upgrade is the workbenches. RHOAI 3.x changes the workbench authentication model (oauth-proxy to kube-rbac-proxy) and routing (Routes to Gateway API). Out-of-the-box images are bumped by the operator, but *custom* (BYON) images and out-of-date image tags need attention -and you have both on this cluster.

In this module you'll discover the custom image, learn exactly what has to change to make a custom image work in 3.x, swap in a pre-built image that already has those changes, bump out-of-date tags, then stop every workbench so it's ready for the upgrade.

== Learning objectives

By the end of this module, you'll be able to:

* Discover custom (BYON) workbench images registered in the dashboard
* Describe the Dockerfile and configuration changes a custom image needs for 3.x
* Re-register a custom ImageStream to point at a rebuilt, 3.x-ready image
* Bump out-of-date image tags and stop every workbench before the upgrade

== Exercise 5.2: Why this image breaks in 3.x

A custom workbench image built for 2.x makes two assumptions that 3.x breaks:

. *Authentication.* In 2.x, each workbench pod ran an `oauth-proxy` sidecar tightly coupled to the internal OpenShift OAuth server. In 3.x, the platform injects `kube-rbac-proxy` instead. An image that bakes in oauth-proxy configuration fights the platform and produces redirect loops.
. *Routing.* In 2.x, workbenches were exposed through OpenShift Routes. In 3.x they're exposed through Kubernetes Gateway API with *path-based* routing, where every workbench lives under a path prefix carried in the `NB_PREFIX` environment variable. An image whose web server ignores `NB_PREFIX` -or strips it from redirects -produces "page not found" once it's behind the Gateway.

The fix is a *rebuild* of the image, not a configuration tweak on the cluster. Red Hat does not auto-rebuild custom images -this is an application-owner task.

== Exercise 5.3: How the image was updated for 3.x

The rebuild has already been done for you and the result is published at `quay.io/hayesphilip/rhoai-workbench-gw:3.x`. This exercise walks through *exactly* what changed in the Containerfile so you can do the same for your own images. Both the original 2.x BYON Containerfile (`examples/custom-workbench-2x/Containerfile`) and the rebuilt 3.x one (`examples/custom-workbench-gw/Containerfile`) ship with the workshop repo.

=== The two changes that make a Jupyter image 3.x-ready

. *No oauth-proxy.* RHOAI 3.x injects `kube-rbac-proxy` at the platform level. Any `oauth-proxy` binary, config, or sidecar wiring in the image must be deleted -the image only needs to serve the notebook on its own port. The 2.x image relied on a platform-injected oauth-proxy sidecar plus the upstream `start-notebook.sh` entrypoint, which is exactly what produces redirect loops under 3.x.
. *`NB_PREFIX`-aware entrypoint.* The notebook controller injects the workbench's path prefix in `NB_PREFIX` (e.g. `/notebook/<ns>/<name>`). Under 3.x, workbenches are exposed through Kubernetes Gateway API with path-based routing, so the notebook server must serve under that prefix *and* must not strip it from redirects. The fix for a Jupyter image is to pass `NB_PREFIX` to `--ServerApp.base_url` at launch.

=== The Containerfile diff

The lab's 2.x and 3.x Containerfiles are otherwise identical (same `FROM`, same packages, same UID workaround). The only meaningful changes:

[source,diff]
----
  FROM quay.io/jupyter/scipy-notebook:lab-4.2.5

  LABEL io.opendatahub.notebooks="true" \
        io.openshift.expose-services="8888:http" \
        ...

  USER root
  RUN pip install --no-cache-dir 'mlflow~=2.9' 'onnxruntime~=1.17'
  RUN chgrp -R 0 /home/jovyan && chmod -R g=u /home/jovyan

+ # Install the RHOAI-aware entrypoint that honors NB_PREFIX.
+ COPY start-rhoai-notebook.sh /usr/local/bin/start-rhoai-notebook.sh
+ RUN chmod +x /usr/local/bin/start-rhoai-notebook.sh

  USER ${NB_UID}
  EXPOSE 8888
- # (no ENTRYPOINT — inherits upstream start-notebook.sh which ignores NB_PREFIX)
+ ENTRYPOINT ["/usr/local/bin/start-rhoai-notebook.sh"]
----

The entrypoint script itself is short -its whole job is launching Jupyter with the controller-supplied prefix as the base URL:

[source,bash]
----
#!/bin/bash
set -euo pipefail

exec jupyter lab \
  --ServerApp.base_url="${NB_PREFIX:-/}" \
  --ServerApp.ip=0.0.0.0 \
  --ServerApp.port=8888 \
  --ServerApp.token='' \
  --ServerApp.password='' \
  --ServerApp.allow_origin='*' \
  --ServerApp.open_browser=False \
  --ServerApp.quit_button=False
----

The Containerfile and entrypoint script are in `examples/custom-workbench-gw/` in the workshop repo -use them as the starting template for your own custom images.

=== Build + push (reference only — already done for you)

[source,bash]
----
cd examples/custom-workbench-gw/
podman build --platform linux/amd64 -t quay.io/hayesphilip/rhoai-workbench-gw:3.x -f Containerfile .
podman push quay.io/hayesphilip/rhoai-workbench-gw:3.x
----

[NOTE]
====
*Three gotchas worth knowing:*

* *Platform* — the `--platform linux/amd64` flag matters. Lab clusters are amd64; a default `podman build` on Apple Silicon produces an arm64 image that fails on the cluster with `exec /usr/bin/tini: Exec format error`.
* *Base image* — production builds should rebase on UBI9 + Python from `registry.redhat.io`. The lab uses the community `quay.io/jupyter/scipy-notebook` base, which is the pragmatic path most BYON authors take.
* *NGINX-fronted images* (code-server, RStudio) need two extra fixes that Jupyter doesn't: keep `NB_PREFIX` in every `Location:` redirect header, and serve `/api` as a 302 instead of an inline rewrite to avoid 403s.
====

== Exercise 5.4: Re-register the ImageStream with the 3.x-ready image

Point the custom ImageStream at the rebuilt image so any workbench created from it uses the 3.x-compatible version. Import the new image as a tag:

[.console-input]
[source,role="execute"]
----
oc tag quay.io/hayesphilip/rhoai-workbench-gw:3.x \
  custom-scipy-notebook:3.x \
  -n redhat-ods-applications \
  --reference-policy=local
----

Update the BYON registration annotation so the dashboard shows the new image URL:

[.console-input]
[source,role="execute"]
----
oc annotate imagestream custom-scipy-notebook -n redhat-ods-applications \
  opendatahub.io/notebook-image-url="quay.io/hayesphilip/rhoai-workbench-gw:3.x" --overwrite
----

`--reference-policy=local` pins the ImageStream to the cluster's internal registry. If you host your custom image there, also grant `system:image-puller` to the workbench ServiceAccount for `redhat-ods-applications`.

=== Verify

[.console-input]
[source,role="execute"]
----
oc get imagestream custom-scipy-notebook -n redhat-ods-applications \
  -o jsonpath='{range .status.tags[*]}{.tag}{"\n"}{end}'
----

✓ A `3.x` tag now exists on the `custom-scipy-notebook` ImageStream

== Exercise 5.5: Bump out-of-date image tags

Some out-of-the-box images break on old tags under 3.x. The most common is `code-server`, which must be on the `2025.2` tag. Find any workbench on an older tag:

[.console-input]
[source,role="execute"]
----
oc get notebooks -A -o json | jq -r '.items[]
  | select(.spec.template.spec.containers[0].image | test("code-?server"))
  | "\(.metadata.namespace)/\(.metadata.name)  \(.spec.template.spec.containers[0].image)"'
----

If any are on `2025.1` or older, bump every one in a single loop -no per-notebook copy/paste, no placeholders to fill in:

[.console-input]
[source,role="execute"]
----
oc get notebooks -A -o json \
  | jq -r '.items[]
      | select(.spec.template.spec.containers[0].image | test("code-?server"))
      | select(.spec.template.spec.containers[0].image | test(":2025\\.[01]$"))
      | "\(.metadata.namespace)/\(.metadata.name)"' \
  | while read -r nb; do
      NS="${nb%/*}"; NAME="${nb##*/}"
      echo "Bumping $NS/$NAME to code-server-notebook:2025.2"
      oc patch notebook "$NAME" -n "$NS" --type=json \
        -p='[{"op":"replace","path":"/spec/template/spec/containers/0/image","value":"image-registry.openshift-image-registry.svc:5000/redhat-ods-applications/code-server-notebook:2025.2"}]'
    done
----

For this lab the loop will hit `workbenches-regular/codeserver-2025-1` and bump it. If your shell paste truncates the multi-line loop, paste it inside `bash -c '...'` to keep zsh from re-parsing line-by-line.

[NOTE]
====
*RStudio is not in this lab's install, but is common in production.* RStudio workbenches are built per-cluster from a BuildConfig, must use the `latest` tag, and need a new build *after* the upgrade. If your cluster has RStudio workbenches, plan that work into the post-upgrade window. Jupyter-based images are generally compatible, but bumping to `2025.2` is recommended.
====

=== Verify

[.console-input]
[source,role="execute"]
----
oc get notebooks -A -o json | jq -r '.items[]
  | select(.spec.template.spec.containers[0].image | test("code-?server"))
  | "\(.metadata.name): \(.spec.template.spec.containers[0].image | split(":")[-1])"'
----

✓ Any code-server workbench is on tag `2025.2` (or there are none on this cluster)

== Exercise 5.6: Stop every workbench

All workbenches must be stopped before the upgrade -running workbenches built for the 2.x auth model hit redirect loops during the transition. The notebook controller treats the `kubeflow-resource-stopped` annotation as "stopped if present"; the dashboard sets it to a timestamp, so use one here.

Survey what's running (by pod count, the reliable signal):

[.console-input]
[source,role="execute"]
----
for nb in $(oc get notebooks -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}{"\n"}{end}'); do
  ns="${nb%/*}"; name="${nb##*/}"
  pods=$(oc -n "$ns" get pods -l notebook-name="$name" --no-headers 2>/dev/null | wc -l | tr -d ' ')
  [ "$pods" -gt 0 ] && echo "$nb has $pods pod(s) running"
done
----

Stop every workbench:

[.console-input]
[source,role="execute"]
----
STAMP=$(date -u +%Y-%m-%dT%H:%M:%SZ)
for row in $(oc get notebooks -A --no-headers | awk '{print $1"/"$2}'); do
  ns="${row%/*}"; name="${row##*/}"
  oc -n "$ns" annotate notebook "$name" \
    kubeflow-resource-stopped="$STAMP" --overwrite
done
----

=== Verify

[.console-input]
[source,role="execute"]
----
for nb in $(oc get notebooks -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}{"\n"}{end}'); do
  ns="${nb%/*}"; name="${nb##*/}"
  pods=$(oc -n "$ns" get pods -l notebook-name="$name" --no-headers 2>/dev/null | wc -l | tr -d ' ')
  [ "$pods" -gt 0 ] && echo "$nb still has $pods pod(s)"
done
echo "survey complete"
----

[.console-output]
----
survey complete
----

✓ No workbench reports running pods

== Troubleshooting

*Issue*: `oc tag` reports it can't reach the image.

*Solution*: Confirm the image URL is correct and public, or that the cluster has a pull secret for the registry. The lab image is published at `quay.io/hayesphilip/rhoai-workbench-gw:3.x`. If it is in a private registry, add a pull secret to the `redhat-ods-applications` namespace first.

*Issue*: A workbench started from the custom image still hits a redirect loop in 3.x.

*Solution*: The image was not rebuilt correctly -confirm oauth-proxy is fully removed and the server honors `NB_PREFIX`. Use the rebuilt `-gw` image from Exercise 5.4, not the original.

*Issue*: A workbench won't stop -its pod keeps coming back.

*Solution*: Confirm the annotation applied: `oc get notebook <name> -n <ns> -o jsonpath='{.metadata.annotations.kubeflow-resource-stopped}'`. The controller treats *any* non-empty value as stopped -use a timestamp, not the literal `true`.
