= Module 7: Post-upgrade verification
:source-highlighter: rouge
:toc: macro
:toclevels: 2

The operator is on {target-version} and the DSC is `Ready`, but the migration isn't finished. The 3.x reconciler needs a few helper scripts to bring components fully across (ConfigMap state, workbench auth, RayCluster CRs, GuardrailsOrchestrator schema). This module runs those scripts and then sweeps every workload to confirm they survived the upgrade.

== Learning objectives

By the end of this module, you'll be able to:

* Confirm the operator, DSC, and DSCI are healthy on {target-version}
* Run the four required post-upgrade helper scripts (ConfigMap, workbenches, Ray, GuardrailsOrchestrator)
* Verify every workload on the cluster is healthy and the rhai-cli lint is clean

== Exercise 7.2: Restore the inferenceservice-config ConfigMap

In Module 4 you marked `inferenceservice-config` as `managed=false` so the upgrade couldn't redeploy your ISVCs. With the operator on {target-version}, hand control back. The official `managed-inferenceservice-config.sh` helper restores the annotation, restarts the KServe controller, and verifies that no ISVC was silently redeployed (guide §4.9.1).

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -- \
  /opt/rhai-upgrade-helpers/model-serving/after-upgrade/managed-inferenceservice-config.sh \
  -n redhat-ods-applications
----

=== Verify

[.console-input]
[source,role="execute"]
----
oc get configmap inferenceservice-config -n redhat-ods-applications \
  -o jsonpath='managed={.metadata.annotations.opendatahub\.io/managed}{"\n"}'
----

✓ Output is `managed=true`

== Exercise 7.3: Patch the workbenches onto the new auth model

Workbenches still carry 2.x oauth-proxy configuration. Under 3.x they need kube-rbac-proxy or they hit redirect loops on first start. The `rhai-cli` helper patches each stopped workbench in place.

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -- bash -c '
  cd /opt/rhai-upgrade-helpers/workbenches && \
  ./workbench-2.x-to-3.x-upgrade.sh patch --only-stopped --with-cleanup -y
'
----

=== Verify

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -- bash -c '
  cd /opt/rhai-upgrade-helpers/workbenches && \
  ./workbench-2.x-to-3.x-upgrade.sh list --all
'
----

✓ Output reports `Legacy (2.x): 0` and `All workbenches have been migrated`

[NOTE]
====
Custom workbench images that bake in oauth-proxy can't be fixed by this patch -they need an image rebuild against kube-rbac-proxy. The helper flags them; the rebuild is an application-owner task.
====

== Exercise 7.5: Patch GuardrailsOrchestrator

Two official §4.6.2 helpers per GuardrailsOrchestrator: `patch-guardrails-deployment.sh` adds the missing ReadinessProbe; `migrate-gorch-otel-exporter.sh` rewrites the `spec.otelExporter` schema to 3.x. The loop runs both for every GORCH:

[.console-input]
[source,role="execute"]
----
oc get guardrailsorchestrator -A -o json \
  | jq -r '.items[]? | "\(.metadata.namespace) \(.metadata.name)"' \
  | while read -r NS GORCH_NAME; do
      echo "=== $NS/$GORCH_NAME ==="
      oc exec -n rhai-migration rhai-cli-0 -- bash -c \
        "cd /opt/rhai-upgrade-helpers/trustyai && ./patch-guardrails-deployment.sh --gorch-name $GORCH_NAME --namespace $NS --fix"
      oc exec -n rhai-migration rhai-cli-0 -- bash -c \
        "cd /opt/rhai-upgrade-helpers/trustyai && ./migrate-gorch-otel-exporter.sh --namespace $NS --fix"
    done
----

=== Verify

[.console-input]
[source,role="execute"]
----
oc get guardrailsorchestrator -A
----

✓ Each GuardrailsOrchestrator eventually reaches a healthy state (may take 1-2 minutes for the deployment to roll)

== Exercise 7.7: Final assessment

Re-run the rhai-cli lint to confirm the tool sees a healthy cluster on {target-version}. When the current version already equals the target, `rhai-cli lint` short-circuits with a "no checks will be executed" line -that itself is the pass signal: nothing left to lint.

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -- \
  /opt/rhai-cli/bin/rhai-cli lint --target-version 3.3.5 2>&1 \
  | tail -5
----

[.console-output]
----
Environment:
  OpenShift AI version: 3.3.5
  OpenShift version:    4.20.27

Current and target versions are the same (3.3), no checks will be executed.
----

✓ `Current and target versions are the same` -the migration is on {target-version} and there is nothing left for the lint to check

To gauge readiness for the *next* upgrade (3.3 → 3.4), point `--target-version` at 3.4 instead. That's optional -it's a preview of the next migration, not a check on this one:

[.console-input]
[source,role="execute"]
----
oc exec -n rhai-migration rhai-cli-0 -- \
  /opt/rhai-cli/bin/rhai-cli lint --target-version 3.4 \
  | grep -E "Total:|Result:"
----

== Troubleshooting

*Issue*: A model returns `Ready=False` after the upgrade.

*Solution*: Confirm Exercise 7.2 (ConfigMap restore) was applied before model verification. If the issue persists, check the predictor pod's `events` and `logs` (`oc get events -n <ns>`, `oc logs <pod>`).

*Issue*: The Gateway dashboard URL doesn't resolve.

*Solution*: On clusters with an SRE-managed admission webhook restricting NetworkPolicy creation in `openshift-*` namespaces, the GatewayConfig reconciler's NetworkPolicy POST is rejected: `oc patch gatewayconfig default-gateway --type=merge -p '{"spec":{"networkPolicy":{"ingress":{"enabled":false}}}}'`.

*Issue*: `workbench-2.x-to-3.x-upgrade.sh list --all` reports `Failed: N`.

*Solution*: Inspect each with `oc describe notebook <name> -n <ns>`. The most common cause is a custom workbench image that still embeds oauth-proxy and must be rebuilt for kube-rbac-proxy.
